第一课 · Lesson 01

LangGraph 核心概念：StateGraph 入门

从零理解 State、Node、Edge 三大核心概念，构建第一个 LangGraph 应用，理解为什么要从 LangChain 迁移到 LangGraph。

State 共享状态 Node 节点函数 Edge 有向边 compile + invoke vs LangChain Chain
1

为什么需要 LangGraph？

你已经学过 LangChain，知道 Chain 可以把多个组件串联起来。但 Chain 有一个根本限制：只能线性执行，无法做条件分支、循环或并行。随着 Agent 越来越复杂，这个限制变得越来越难以接受。

对比维度LangChain ChainLangGraph Graph
执行方式固定线性管道 A→B→C有向图，支持任意连接
条件分支❌ 不支持✅ add_conditional_edges
循环执行❌ 不支持✅ 节点可以指向已执行节点
状态管理各组件独立，手动传递全局 State，所有节点共享
并行执行❌ 不支持✅ 支持（高级特性）
人工介入❌ 不支持✅ interrupt_before/after
适合场景简单 RAG、单轮问答Agent、多步推理、对话系统
💡 一句话理解
LangChain Chain 是"流水线"，LangGraph 是"状态机"。当你的 AI 应用需要根据结果做决策、需要循环推理、或需要人工介入时，就该用 LangGraph。
2

核心概念一：State（共享状态）

State 是 LangGraph 中最重要的概念。它是一个 Python TypedDict，充当整个 Graph 的"共享内存"——所有节点都从这里读数据、把结果写回这里。

from typing import TypedDict

class State(TypedDict):
    question: str    # 用户输入的问题
    answer: str      # LLM 的回答
    word_count: int  # 回答的字数统计
📦
只放需要跨节点传递的数据节点内部的临时变量不需要放进 State，只有"A 节点写入、B 节点读取"的数据才放进去。
🔑
用 TypedDict 提供类型提示IDE 能自动补全字段名，减少拼写错误。LangGraph 也用它做运行时校验。
🔄
字段默认行为是"覆盖"节点返回 {"answer": "新值"} 会直接覆盖旧值。需要"追加"行为时，用 Reducer（第四课讲解）。
⚠️ 与 LangChain 的区别
LangChain 的 RunnablePassthrough 把数据逐级传递，上下游耦合紧密。LangGraph 的 State 是全局共享的，任何节点都能读任何字段，解耦更彻底。
3

核心概念二：Node（节点）

节点就是普通的 Python 函数。签名固定：接收完整的 State，返回一个字典（只包含要更新的字段）。

def llm_node(state: State) -> dict:
    # 从 State 读取输入
    response = llm.invoke([HumanMessage(content=state["question"])])
    # 只返回要更新的字段
    return {"answer": response.content}

def count_node(state: State) -> dict:
    return {"word_count": len(state["answer"])}

def print_node(state: State) -> dict:
    print(state["answer"])
    return {}  # 不更新任何字段，返回空字典
1️⃣
参数只有一个：state函数接收当前完整的 State 快照，不能有其他必要参数。如需注入额外依赖（如数据库连接），可以用闭包或类方法。
2️⃣
返回值是 dict，只含要更新的字段LangGraph 会把返回的字典"合并"回 State 中，未返回的字段保持原值不变。返回 {} 表示不修改任何字段。
3️⃣
节点本身无状态所有状态都在 State 里，节点函数可以在不同 Graph 里复用。建议用动词命名，如 classify_intent、call_llm。
4

核心概念三：Edge（边）

Edge 定义了节点之间的执行顺序。本课介绍固定边（无条件跳转），下一课学习条件边。

from langgraph.graph import END

# 固定边：llm 执行完后，始终跳转到 count
graph_builder.add_edge("llm", "count")

# 指向内置终止符 END，表示 Graph 在此结束
graph_builder.add_edge("print", END)
Edge 类型方法说明
固定边（本课）add_edge(A, B)A 执行完后始终去 B
条件边（第2课）add_conditional_edges(A, fn)根据函数返回值动态决定去哪
入口点set_entry_point(node)Graph 从哪个节点开始
终止add_edge(node, END)Graph 在此停止并返回 State
🔮 END 的含义
END 是 LangGraph 内置终止符，从 langgraph.graph 导入。一个 Graph 可以有多个节点都指向 END（多出口），invoke() 在第一个到达 END 的路径结束时返回。
5

构建与编译 Graph

LangGraph 采用"先定义、后编译"的模式。compile() 会对 Graph 做合法性校验，确保所有节点都可达、入口已设置等。

# Step 1：创建 GraphBuilder，指定 State 类型
graph_builder = StateGraph(State)

# Step 2：注册节点（"名称" → 函数）
graph_builder.add_node("llm",   llm_node)
graph_builder.add_node("count", count_node)
graph_builder.add_node("print", print_node)

# Step 3：连接边
graph_builder.add_edge("llm",   "count")
graph_builder.add_edge("count", "print")
graph_builder.add_edge("print", END)

# Step 4：设置入口
graph_builder.set_entry_point("llm")

# Step 5：编译（锁定配置，校验合法性）
graph = graph_builder.compile()

# Step 6：运行（传入初始 State，返回最终 State）
result = graph.invoke({
    "question": "用一句话解释什么是人工智能",
    "answer": "",
    "word_count": 0,
})
6

核心知识点总结

⭐
State 是唯一的数据载体节点之间不直接传参，全靠读写同一个 State 字典。类比：State 就像数据库，节点是对其进行读写的函数。
⭐
节点只返回要更新的字段返回 {"answer": "xxx"} 只更新 answer，其他字段保持不变。返回 {} 代表不修改任何东西。
⭐
END 是内置终止符from langgraph.graph import END，指向它的边就是 Graph 的出口，一个 Graph 可以有多个出口。
⭐
compile() 锁定并校验配置如果 Graph 配置有误（如入口未设置、节点不可达），在 compile() 阶段就会报错，而不是运行时才出问题。
⭐
invoke() 同步执行并返回最终 State传入初始 State 字典，返回 Graph 执行完毕时的完整 State。如需流式输出，可用 graph.stream()。

第二课 · Lesson 02

条件路由：让 Graph 动态决策

掌握 add_conditional_edges，让 Graph 根据 State 内容动态决定走哪条路——这是构建真正 Agent 的关键能力。

add_conditional_edges 路由函数 Router 意图分类 扇出-扇入模式
1

什么是条件路由？

第一课的 Graph 中，每条边都是"固定的"——A 执行完一定去 B。条件路由让这条边变成"动态的"——A 执行完后，根据当前 State 的内容，决定去 B、C 还是 D。

对比add_edge（固定边）add_conditional_edges（条件边）
跳转目标固定，永远去同一个节点动态，由路由函数决定
判断时机无需判断每次执行时调用路由函数
适用场景固定流程分类、决策、Agent 循环
代码示例add_edge("A", "B")add_conditional_edges("A", fn)

本课场景：用户输入问题 → LLM 先判断问题类型（意图分类）→ 根据类型路由到不同处理节点

📌 场景说明
  • 问候类（greeting）→ greeting_handler：轻松友好的回复
  • 技术类（technical）→ technical_handler：专业详细的回复
  • 通用类（general）→ general_handler：通用简洁的回复
2

路由函数（Router Function）

路由函数是条件边的核心。它的规则很简单：

def route_by_intent(state: State) -> str:
    # 规则1：接收 State 作为唯一参数
    intent = state["intent"]

    # 规则2：返回下一个节点的"名称字符串"
    if intent == "greeting":
        return "greeting_handler"
    elif intent == "technical":
        return "technical_handler"
    else:
        return "general_handler"
    # 规则3：也可以返回 END，代表直接结束
✅
路由函数只做"决策"，不做"处理"它只读取 State，返回节点名称字符串，不修改 State。真正的业务逻辑放在各个 handler 节点里——这是关注点分离的体现。
✅
返回值是节点名称字符串返回的字符串必须与 add_node() 注册的节点名称完全一致，否则运行时报错。也可以返回内置的 END。
3

add_conditional_edges 用法

# 用法一：最简形式（路由函数直接返回节点名）
graph_builder.add_conditional_edges(
    "classify",      # 从哪个节点出发
    route_by_intent,  # 路由函数
)

# 用法二：带映射表（更清晰，推荐）
graph_builder.add_conditional_edges(
    "classify",
    route_by_intent,
    {                             # 映射表：路由函数返回值 → 实际节点名称
        "greeting_handler":  "greeting_handler",
        "technical_handler": "technical_handler",
        "general_handler":   "general_handler",
    }
)

# 用法三：映射表中的值可以与 key 不同（用于别名）
graph_builder.add_conditional_edges(
    "classify",
    route_by_intent,
    {
        "greeting":  "greeting_handler",  # key 是路由函数返回值，value 是节点名
        "technical": "technical_handler",
        "general":   "general_handler",
    }
)
⚠️ 注意：映射表的作用
当路由函数的返回值与节点名称不一样时，必须提供映射表。比如路由函数返回 "greeting"，但节点名是 "greeting_handler"，就需要用映射表做转换。如果返回值和节点名完全相同，映射表可以省略。
4

常见疑问：set_entry_point 和 add_conditional_edges 都写了 "classify"，有什么区别？

这是一个非常好的问题。在原始代码里，两行都出现了 "classify"，容易让人误以为它们是同一件事：

graph_builder.set_entry_point("classify")          # 为什么这里也是 classify？
graph_builder.add_conditional_edges("classify", ...)  # 这里也是 classify？

它们是完全不同的两件事，只是碰巧指向了同一个节点名。

set_entry_point("classify")add_conditional_edges("classify", ...)
作用告诉 Graph：invoke() 时
第一步从哪个节点开始		告诉 Graph：classify 节点执行完之后
往哪里走
影响范围全局入口，只设置一次仅定义 classify 节点的出边
类比地图上标注"游客从A门进入"A门内部的指路牌："左走/直走/右走"

为了让区别更清晰，代码里加入了一个 preprocess_node（预处理节点），让入口节点和条件边出发节点变成两个不同的节点：

# 入口改为 preprocess，不再是 classify
graph_builder.set_entry_point("preprocess")

# preprocess 执行完后，固定跳到 classify
graph_builder.add_edge("preprocess", "classify")

# classify 执行完后，条件分支（与入口无关）
graph_builder.add_conditional_edges("classify", route_by_intent, {...})

这样执行顺序就变成：

invoke() 开始 │ ▼ [preprocess] ← set_entry_point("preprocess") 的作用：从这里开始 │ ▼ add_edge("preprocess", "classify") 固定跳转 │ ▼ [classify] ← add_conditional_edges("classify", ...) 的作用：从这里出发做条件分支 │ ├──▶ [greeting_handler] ├──▶ [technical_handler] └──▶ [general_handler]
✅ 结论
  • set_entry_point = "invoke 时第一脚踩在哪个节点"，全局只设置一次
  • add_conditional_edges 第一个参数 = "这个节点跑完之后，用路由函数决定下一步"，是在定义某个节点的出边
  • 原始代码两处都写 "classify"，只是因为 classify 恰好也是入口节点。加了 preprocess 之后，两个参数就完全不同了，职责清晰可见。
5

完整代码讲解（02_conditional_routing.py）

# ── State 定义 ────────────────────────────────────────────
class State(TypedDict):
    question: str  # 用户问题
    intent: str    # 意图分类结果：greeting / technical / general
    answer: str    # 最终回答

# ── 节点0：预处理（入口节点） ─────────────────────────────
# 存在目的：让 set_entry_point 和 add_conditional_edges 指向不同节点
def preprocess_node(state: State) -> dict:
    cleaned = state["question"].strip()
    return {"question": cleaned}

# ── 节点1：意图分类 ───────────────────────────────────────
def classify_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content=(
            "你是意图分类器，只返回以下三个词之一：\n"
            "greeting / technical / general"
        )),
        HumanMessage(content=state["question"]),
    ])
    intent = response.content.strip().lower()
    # 防御性处理：LLM 没按格式返回时，默认 general
    if intent not in ("greeting", "technical", "general"):
        intent = "general"
    return {"intent": intent}

# ── 节点2a/b/c：三个不同的处理节点 ──────────────────────
def greeting_handler(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content="用轻松愉快的语气回复，2句话以内"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

# technical_handler 和 general_handler 类似，换 System Prompt 即可

# ── 路由函数 ────────────────────────────────────────────
def route_by_intent(state: State) -> str:
    intent = state["intent"]
    if intent == "greeting":   return "greeting_handler"
    elif intent == "technical": return "technical_handler"
    else:                       return "general_handler"

# ── 构建 Graph ───────────────────────────────────────────
graph_builder = StateGraph(State)
graph_builder.add_node("preprocess",        preprocess_node)  # 新增
graph_builder.add_node("classify",          classify_node)
graph_builder.add_node("greeting_handler",  greeting_handler)
graph_builder.add_node("technical_handler", technical_handler)
graph_builder.add_node("general_handler",   general_handler)
graph_builder.add_node("print_result",      print_result)

# ★ set_entry_point：入口是 preprocess，invoke() 从这里开始
graph_builder.set_entry_point("preprocess")

# preprocess 执行完后，固定跳到 classify
graph_builder.add_edge("preprocess", "classify")

# ★ add_conditional_edges：classify 执行完后做条件分支（与入口无关）
graph_builder.add_conditional_edges(
    "classify", route_by_intent,  # "classify" 指的是"从哪个节点出发"，不是入口
    {"greeting_handler": "greeting_handler",
     "technical_handler": "technical_handler",
     "general_handler": "general_handler"}
)

# 扇入：三条路径汇聚到同一个 print_result
graph_builder.add_edge("greeting_handler",  "print_result")
graph_builder.add_edge("technical_handler", "print_result")
graph_builder.add_edge("general_handler",   "print_result")
graph_builder.add_edge("print_result", END)

graph = graph_builder.compile()
6

核心知识点总结

⭐
add_edge vs add_conditional_edgesadd_edge 是固定路径，永远从 A 到 B；add_conditional_edges 是动态路径，根据路由函数的返回值决定去哪个节点。
⭐
set_entry_point 和 add_conditional_edges 的第一个参数是完全不同的两件事set_entry_point("X") = invoke() 时从 X 节点开始执行（全局入口）；add_conditional_edges("X", fn) = X 节点执行完之后，用 fn 决定下一步（X 节点的出边）。两者碰巧写同一个节点名，只是因为该节点恰好也是入口。加入 preprocess 节点后，入口变成 preprocess，条件边依然从 classify 出发，两者的职责就完全分开了。
⭐
路由函数：只做决策，不做处理路由函数只读取 State 并返回节点名称字符串，真正的业务逻辑放在各个 handler 节点里。关注点分离让代码更清晰。
⭐
扇出-扇入（Fan-out / Fan-in）模式从一个节点出发，分叉到多个节点（扇出）；多个节点执行完后，汇聚到同一个后续节点（扇入）。这是 Graph 中最常见的结构模式。
⭐
LLM 路由的防御性处理用 LLM 做分类时，它可能不按格式返回。一定要加 fallback 逻辑（如 if intent not in (...): intent = "general"），避免路由函数返回不存在的节点名导致崩溃。
⭐
映射表是可选的但推荐写出来提供映射表 {"路由返回值": "节点名"} 可以让 LangGraph 在 compile 阶段就检测出"返回了未注册的节点名"这类错误，而不是等到运行时才报错。

第三课 · Lesson 03

ReAct Agent：让 LLM 循环调用工具

掌握 ReAct 模式——LLM 推理、调用工具、观察结果、再推理，循环直到得出最终答案。这是构建真正能"干活"的 Agent 的核心结构。

Reducer / add_messages @tool 装饰器 bind_tools() ToolNode（内置） tools_condition（内置） 循环 Graph 结构
1

ReAct 模式是什么？

ReAct 是 Reason + Act 的缩写。LLM 不是一次性给出答案，而是循环执行以下步骤：

🧠
Reason（推理）LLM 思考：当前问题需要什么信息？该调用哪个工具？参数是什么？
🔧
Act（行动）LLM 声明调用某个工具，LangGraph 的 ToolNode 负责实际执行，返回工具结果。
👀
Observe（观察）工具结果作为新消息追加到 messages 列表，LLM 在下一轮能看到这个结果，再决定下一步。

循环会一直执行，直到 LLM 认为已经有足够信息，不再调用工具，直接给出最终文字回答为止。

💡 与前两课的区别
前两课的 Graph 都是"直线+分支"，每个节点最多执行一次。ReAct Agent 引入了循环——tools 节点执行完后，用 add_edge("tools", "agent") 把执行流导回 agent 节点，形成多轮循环。
2

Reducer：add_messages（追加而不是覆盖）

前两课的 State 字段是"覆盖"语义：节点返回 {"answer": "新值"} 会直接替换旧值。但消息历史需要"追加"——每轮对话的新消息要保留，不能覆盖之前的历史。

LangGraph 用 Reducer 解决这个问题：通过 Annotated 给字段指定合并函数。

from typing import Annotated, TypedDict
from langgraph.graph.message import add_messages
from langchain_core.messages import BaseMessage

class State(TypedDict):
    # Annotated 第2个参数 add_messages = "合并时追加，不覆盖"
    messages: Annotated[list[BaseMessage], add_messages]
场景普通字段带 add_messages 的字段
节点返回 {"messages": [新消息]}整个列表被替换为 [新消息]新消息追加到列表末尾
执行3轮后只保留最后一条保留全部3轮的消息
适用场景单次结果（answer, intent…）对话历史、工具调用记录
⚠️ 消息数量的变化
代码里打印了 消息数：1 → 消息数：3，这就是 add_messages 在起作用：第一轮1条（用户问题），工具调用后变成3条（用户问题 + AI的tool_calls声明 + 工具返回结果），第二轮 agent 看到的就是这3条完整历史。
3

工具定义与 bind_tools()

用 @tool 装饰器把普通函数变成 LLM 可调用的工具。LangGraph 会自动提取函数名、docstring、参数类型作为工具描述传给 LLM。

from langchain_core.tools import tool

@tool
def add(a: int, b: int) -> int:
    """计算两个整数的和"""  # docstring 就是工具描述，LLM 靠这个判断何时调用
    return a + b

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气情况"""
    return weather_data.get(city, f"暂无数据")

tools = [add, multiply, get_weather]

# bind_tools：把工具注册到 LLM，LLM 的回复可能包含 tool_calls
llm_with_tools = llm.bind_tools(tools)
📝
docstring 至关重要LLM 依靠 docstring 判断该调用哪个工具。描述要清晰准确，否则 LLM 可能调错工具或不调用工具。
🔗
bind_tools() 后 LLM 的回复有两种可能普通文字回答（content 有值，tool_calls 为空）；或工具调用声明（tool_calls 有值，content 为空）。
4

内置组件：ToolNode 与 tools_condition

LangGraph 提供了两个内置组件，让你不需要手写工具执行和条件判断逻辑：

from langgraph.prebuilt import ToolNode, tools_condition

# ToolNode：内置节点，自动执行工具调用
# 做了什么：读取最后一条 AIMessage 的 tool_calls → 执行对应函数 → 包装成 ToolMessage 追加
graph_builder.add_node("tools", ToolNode(tools))

# tools_condition：内置路由函数
# 做了什么：检查最后一条消息是否有 tool_calls
#   有 tool_calls → 返回 "tools"（去执行工具）
#   没有         → 返回 END（LLM 已经给出最终答案）
graph_builder.add_conditional_edges("agent", tools_condition)
组件类型作用等效手写代码
ToolNode(tools)节点 读取 tool_calls，执行工具，追加 ToolMessage 需要手写20+行代码
tools_condition路由函数 有 tool_calls → "tools"；无 → END lambda s: "tools" if s["messages"][-1].tool_calls else END
5

循环结构解析

ReAct Agent 的 Graph 结构只有两个节点，但通过一条回边形成循环：

graph_builder.set_entry_point("agent")

graph_builder.add_conditional_edges("agent", tools_condition)
# tools_condition 返回 "tools" 或 END

graph_builder.add_edge("tools", "agent")
# ↑ 这条回边是循环的关键：tools 执行完后回到 agent

循环的终止条件由 LLM 自己决定——当 LLM 认为已有足够信息，回复中不包含 tool_calls，tools_condition 就返回 END，循环结束。

🔮 并行工具调用
从运行结果可以看到，问"北京和上海天气"时，LLM 同时发出了两个 get_weather 调用，ToolNode 并行执行后一起追加到 messages。这是 LLM 的优化行为——能一次解决的不分两次调用。
6

完整代码讲解（03_react_agent.py）

# ── State（Reducer 版本）─────────────────────────────────
class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]
    # 没有 answer 字段！最终答案就是 messages 最后一条

# ── 工具定义 ─────────────────────────────────────────────
@tool
def add(a: int, b: int) -> int:
    """计算两个整数的和"""
    return a + b

@tool
def multiply(a: int, b: int) -> int:
    """计算两个整数的乘积"""
    return a * b

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气情况"""
    return weather_data.get(city, f"暂无数据")

tools = [add, multiply, get_weather]
llm_with_tools = llm.bind_tools(tools)  # LLM 注册工具

# ── Agent 节点 ────────────────────────────────────────────
def agent_node(state: State) -> dict:
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}  # add_messages 会追加，不覆盖

# ── 构建 Graph ────────────────────────────────────────────
graph_builder = StateGraph(State)
graph_builder.add_node("agent", agent_node)
graph_builder.add_node("tools", ToolNode(tools))  # 内置节点

graph_builder.set_entry_point("agent")
graph_builder.add_conditional_edges("agent", tools_condition)  # 内置路由
graph_builder.add_edge("tools", "agent")  # 循环回边

graph = graph_builder.compile()

# ── 运行 ──────────────────────────────────────────────────
result = graph.invoke({
    "messages": [HumanMessage(content="先算 6×7，再加 100")]
})
print(result["messages"][-1].content)  # 取最后一条消息即最终答案

核心知识点总结

⭐
Reducer（add_messages）：追加而不是覆盖用 Annotated[list[BaseMessage], add_messages] 声明字段，节点每次返回的新消息都会追加到列表末尾，保留完整的对话+工具调用历史。
⭐
@tool + docstring = 工具描述LLM 依靠 docstring 判断何时调用哪个工具。docstring 要清晰，参数要有类型注解，否则 LLM 可能误用工具。
⭐
bind_tools() 让 LLM "知道"工具存在未 bind 的工具 LLM 完全不知道，bind 后 LLM 的回复可能包含 tool_calls（工具调用声明），由 ToolNode 负责实际执行。
⭐
add_edge("tools", "agent") 是循环的关键这条回边让 tools 执行完后重新进入 agent，形成多轮循环。没有它，工具执行完 Graph 就终止了，LLM 看不到工具结果。
⭐
循环终止由 LLM 自主决定tools_condition 检查最后一条消息是否有 tool_calls。LLM 认为不需要再调用工具时，返回纯文字回复，tools_condition 路由到 END，循环自然结束。
⭐
最终答案是 messages 的最后一条与前两课不同，ReAct Agent 没有单独的 answer 字段。运行结束后取 result["messages"][-1].content 即为最终答案。

第四课 · Lesson 04

Checkpointer：给 Agent 加上持久化记忆

第三课的 Agent 每次 invoke 都从零开始，不知道上次说了什么。本课用 Checkpointer 保存每步 State 快照，让同一 thread_id 下的多次对话共享完整历史。

MemorySaver compile(checkpointer) thread_id 会话标识 config 运行时参数 多会话隔离 get_state() 查看历史
1

无记忆 Agent 的问题

第三课的 Agent 每次 invoke 都是全新的——它不知道上一次对话说了什么：

# 第三课的写法（无记忆）
graph.invoke({"messages": [HumanMessage("3×4 等于多少？")]})
# → AI：12

graph.invoke({"messages": [HumanMessage("把刚才的结果加上 8")]})
# → AI：不知道"刚才的结果"是什么，无法回答

原因是：每次 invoke 传入的是全新的初始 State，之前对话的 messages 没有被保存，也没有被传入。

💡 Checkpointer 解决了什么
Checkpointer 在每个节点执行后自动保存当前 State 快照。下次同一 thread_id 调用时，LangGraph 先从 Checkpointer 恢复上次的 State，再把新消息追加进去，LLM 就能看到完整的对话历史。
2

Checkpointer 原理

Checkpointer 的工作流程如下：

第1次 invoke(thread_id="A", "3×4=?") ├── 查找 thread_id="A" 的历史快照 → 空（第一次） ├── 初始 State = { messages: [Human("3×4=?")] } ├── 正常执行 Graph（agent → tools → agent → END） └── 保存最终 State 快照到 thread_id="A" 第2次 invoke(thread_id="A", "把刚才的结果加上8") ├── 查找 thread_id="A" 的历史快照 → 找到！ ├── 恢复 State = { messages: [Human, AI(tool), Tool, AI("12")] } ├── 把新消息追加进去（add_messages） ├── State = { messages: [Human, AI, Tool, AI, Human("加上8")] } ├── LLM 看到完整历史，知道"刚才的结果"是 12 └── 保存最新快照
Checkpointer 类型存储位置适用场景
MemorySaver进程内存开发测试，重启后丢失
SqliteSaver本地 SQLite 文件单机持久化
PostgresSaverPostgreSQL 数据库生产环境，多进程共享
✅ 三种 Checkpointer 接口完全相同
只需在 compile(checkpointer=...) 处替换，其他代码一行不用改。开发用 MemorySaver，上线换 PostgresSaver，零成本迁移。
3

thread_id 与多会话隔离

thread_id 是会话的唯一标识符，决定了"这次 invoke 属于哪个对话"：

# config 是 LangGraph 运行时参数，每次 invoke 都要传
config_alice = {"configurable": {"thread_id": "user_alice"}}
config_bob   = {"configurable": {"thread_id": "user_bob"}}

graph.invoke({"messages": [HumanMessage("5+6=?")]},  config=config_alice)
# Alice 的会话：知道 5+6=11

graph.invoke({"messages": [HumanMessage("刚才那道题答案？")]}, config=config_bob)
# Bob 的会话：全新，不知道 Alice 问了什么
🔑
thread_id 可以是任意字符串实际项目中通常用用户ID、会话UUID等业务标识。同一用户可以有多个 thread_id（多个独立对话）。
⚠️
忘传 config 是常见错误如果 invoke 时不传 config，Checkpointer 不知道是哪个会话，每次都是全新状态，记忆功能失效。
4

完整代码讲解（04_memory_agent.py）

# State、工具、节点与第三课完全相同，唯一的区别在 compile

from langgraph.checkpoint.memory import MemorySaver

# ★ 第1处变化：创建 Checkpointer
memory = MemorySaver()

# ★ 第2处变化：compile 时注入 checkpointer
graph = graph_builder.compile(checkpointer=memory)

# ★ 第3处变化：invoke 时传入 config（指定 thread_id）
def chat(thread_id: str, user_message: str) -> str:
    config = {"configurable": {"thread_id": thread_id}}
    result = graph.invoke(
        {"messages": [HumanMessage(content=user_message)]},
        config=config,
    )
    return result["messages"][-1].content

# 多轮对话
chat("session_A", "3 乘以 4 等于多少？")    # → 12
chat("session_A", "把刚才的结果再加上 8")    # → 20（记住了上轮结果）
chat("session_A", "我们算了几道题？")        # → 2道（完整历史可见）

# 查看某会话的完整历史
state_snapshot = graph.get_state({"configurable": {"thread_id": "session_A"}})
for msg in state_snapshot.values["messages"]:
    print(msg.__class__.__name__, msg.content[:50])

核心知识点总结

⭐
只需3处改动就能加上记忆① 创建 MemorySaver()；② compile(checkpointer=memory)；③ invoke 时传 config={"configurable": {"thread_id": "xxx"}}。State、节点、边一行不用改。
⭐
Checkpointer 保存的是完整 State，不只是消息如果 State 里有其他字段（如 intent、answer），也会被保存和恢复。这让跨轮次的复杂状态流转成为可能。
⭐
thread_id 决定会话边界相同 thread_id = 同一对话（共享历史）；不同 thread_id = 独立对话（互不影响）。实际项目中用用户ID或会话UUID作为 thread_id。
⭐
get_state() 可以随时查看快照graph.get_state(config) 返回指定会话当前的完整 State，可用于调试、审计或展示对话历史。
⭐
MemorySaver 仅用于开发测试进程重启后所有会话历史丢失。生产环境替换为 SqliteSaver 或 PostgresSaver，接口完全一致，只改 compile 一行代码。

第五课 · Lesson 05

Human-in-the-Loop：人工介入与审核

让 Agent 在执行危险操作前暂停，等待人工审核。人可以批准、拒绝、或修改参数后再继续——这是 AI 安全落地的关键机制。

interrupt_before invoke(None) 恢复 update_state 拒绝/修改 as_node 参数 get_state 查看暂停状态
1

为什么需要人工介入？

前几课的 Agent 一旦 invoke 就会自动执行到底，中途没有任何停顿。这对以下场景是危险的：

⚠️
不可逆操作删除文件、发送邮件、提交数据库事务——一旦执行无法撤回。
⚠️
高风险操作转账、修改权限、调用付费 API——执行前需要人工确认金额、范围是否正确。
⚠️
LLM 可能误解意图用户说"清理一下日志"，LLM 可能理解为删除所有日志文件。让人看一眼再执行更安全。
💡 核心思路
在 工具执行节点之前插入一个暂停点。Agent 完成规划（决定调用什么工具、什么参数）后，Graph 暂停，把控制权交还给外部程序，等待人类做出决定后再继续。
2

interrupt_before 原理

interrupt_before 是 compile() 的参数，指定在哪个节点执行前暂停：

# ★ 与第四课唯一的区别：多了 interrupt_before 参数
graph = graph_builder.compile(
    checkpointer=memory,
    interrupt_before=["tools"],  # 每次即将执行 tools 节点前暂停
)

# 第一次 invoke：Agent 规划完成，在 tools 前暂停，返回当前 State
graph.invoke(
    {"messages": [HumanMessage("给 boss 发一封周报邮件")]},
    config=config,
)
# ↑ 此时 Agent 已经决定调用 send_email，但还没执行
# graph.invoke 提前返回，等待人工指令

# 查看 Agent 准备做什么
state = graph.get_state(config)
pending_tools = state.values["messages"][-1].tool_calls
print(pending_tools)  # [{'name': 'send_email', 'args': {...}}]
⚠️ 前提条件
interrupt_before 必须配合 checkpointer 使用。Checkpointer 负责保存暂停时的完整 State，这样才能在人工决策后从断点恢复。没有 Checkpointer，暂停状态会丢失。
3

常见疑问：hasattr 判断是什么意思？

查看暂停状态时经常看到这段代码：

if hasattr(last_msg, "tool_calls") and last_msg.tool_calls:
    ...

这是两层保险，缺一不可：

1️⃣
hasattr(last_msg, "tool_calls") —— 检查属性是否存在 消息有多种类型：HumanMessage、AIMessage、ToolMessage。 只有 AIMessage 才有 tool_calls 属性，其他类型根本没有这个字段。 直接访问不存在的属性会抛出 AttributeError，所以先用 hasattr 探一下。
2️⃣
last_msg.tool_calls —— 检查值是否非空 即使属性存在，值也可能是空列表 []（LLM 没有调用任何工具，直接文字回答）。 空列表在 Python 中是 False，这一层确保里面真的有工具调用内容。

两个条件合起来的含义："这条消息是 AIMessage，并且 LLM 声明要调用至少一个工具"

更清晰的等价写法（推荐）：

from langchain_core.messages import AIMessage

# isinstance 比 hasattr 更语义化：明确说"这是一条 AI 消息"
if isinstance(last_msg, AIMessage) and last_msg.tool_calls:
    ...
消息类型有 tool_calls 属性？可能的值
HumanMessage❌ 没有—
AIMessage（文字回答）✅ 有[]（空列表）
AIMessage（工具调用）✅ 有[{'name': 'xxx', 'args': {...}}]
ToolMessage❌ 没有—
4

批准：invoke(None, config) 恢复执行

# 人工确认后，传 None 恢复执行
# None 表示：不新增任何消息，从暂停点继续往下走
result = graph.invoke(None, config=config)

# Graph 从暂停点（tools 节点入口）继续执行：
# tools → agent（看到工具结果）→ tools_condition（无 tool_calls）→ END
5

拒绝：update_state 注入取消消息

拒绝的核心思路：把 Checkpointer 里待执行的 AIMessage(tool_calls=[...]) 替换成一条无 tool_calls 的消息，这样 tools_condition 就会路由到 END。

# update_state：直接修改 Checkpointer 中保存的 State 快照
graph.update_state(
    config,
    {"messages": [AIMessage(content="用户拒绝了该操作，已取消。")]},
    as_node="agent",   # 关键：告诉 LangGraph 这次更新视为 agent 节点的输出
)
# as_node="agent" 的作用：
# LangGraph 认为 agent 节点刚刚运行完，接下来走 tools_condition
# 新消息没有 tool_calls → tools_condition 返回 END → 直接结束

result = graph.invoke(None, config=config)
# 最终回复就是我们注入的取消消息
6

修改参数后批准

最强大的用法：人工发现参数有误（如金额过大），直接修改参数后让工具以正确参数执行：

# 获取当前 Agent 计划调用的 AIMessage
state = graph.get_state(config)
last_ai_msg = state.values["messages"][-1]

# 构造修改后的消息（保留 tool_calls 结构，只改参数）
modified_tool_calls = []
for tc in last_ai_msg.tool_calls:
    modified_tc = dict(tc)
    modified_tc["args"] = {**tc["args"], "amount": 100.0}  # 把 999999 改为 100
    modified_tool_calls.append(modified_tc)

modified_msg = AIMessage(
    content=last_ai_msg.content,
    tool_calls=modified_tool_calls,
)

# 替换原始 AIMessage，as_node="agent" 让 Graph 继续走 tools 分支
graph.update_state(config, {"messages": [modified_msg]}, as_node="agent")

# 恢复执行，tools 节点以修改后的参数（amount=100）执行工具
result = graph.invoke(None, config=config)
7

进阶：部分工具需要确认，部分直接执行

interrupt_before 是节点级别的，不是工具级别的。要实现"只拦截危险工具"，需要把工具按安全性拆进两个不同的 ToolNode，只对危险工具节点设置暂停。

内置 tools_condition（单节点）自定义路由（双节点）
工具节点一个 tools 节点装所有工具safe_tool_node + dangerous_tool_node
暂停范围所有工具调用前都暂停只有危险工具节点前暂停
路由函数内置 tools_condition自定义 selective_router
 
# 工具分组
SAFE_TOOLS      = [get_weather, calculate]      # 直接执行
DANGEROUS_TOOLS = [send_email, delete_files]    # 需要审核
DANGEROUS_NAMES = {t.name for t in DANGEROUS_TOOLS}

# 自定义路由：根据 tool_calls 的工具名决定去哪个节点
def selective_router(state):
    from langchain_core.messages import AIMessage
    last = state["messages"][-1]
    if not isinstance(last, AIMessage) or not last.tool_calls:
        return END
    names = {tc["name"] for tc in last.tool_calls}
    if names & DANGEROUS_NAMES:   # 交集非空 = 含有危险工具
        return "dangerous_tool_node"
    return "safe_tool_node"

# 构建 Graph
graph_builder.add_node("agent",               agent_node)
graph_builder.add_node("safe_tool_node",      ToolNode(SAFE_TOOLS))
graph_builder.add_node("dangerous_tool_node", ToolNode(DANGEROUS_TOOLS))
graph_builder.set_entry_point("agent")
graph_builder.add_conditional_edges("agent", selective_router)
graph_builder.add_edge("safe_tool_node",      "agent")
graph_builder.add_edge("dangerous_tool_node", "agent")

graph = graph_builder.compile(
    checkpointer=memory,
    interrupt_before=["dangerous_tool_node"],  # 只拦截危险工具节点
)
agent │ ├─ tool_calls 全是安全工具 ──▶ [safe_tool_node] 直接执行，无暂停 │ │ │ └──▶ agent（循环） │ ├─ tool_calls 含有危险工具 ──▶ [dangerous_tool_node] ← interrupt_before 在这里 │ │（人工审核后才执行） │ └──▶ agent（循环） │ └─ 无 tool_calls ──▶ END 运行结果验证： "查天气 + 计算" → safe_tool_node → 无暂停，直接跑完 "查天气 + 发邮件" → 含 send_email → dangerous_tool_node → 暂停等审核
✅ 核心设计原则
interrupt_before 是节点级别的暂停，不是工具级别的。要实现工具粒度的控制，就把工具按安全性分配到不同节点，然后只对危险节点设置 interrupt。
9

核心知识点总结

⭐
hasattr + tool_calls 的两层判断hasattr(msg, "tool_calls") 检查属性是否存在（只有 AIMessage 有），msg.tool_calls 检查值是否非空。更清晰的写法：isinstance(msg, AIMessage) and msg.tool_calls。
⭐
interrupt_before 必须配合 CheckpointerCheckpointer 保存暂停时的完整 State，人工决策后才能从断点恢复。两者缺一不可。
⭐
invoke(None) = 从暂停处继续，不新增消息传 None 告诉 LangGraph：不要追加新消息，直接从上次暂停的节点入口继续执行。
⭐
update_state 直接修改 Checkpointer 的快照可以修改任何 State 字段。as_node="agent" 指定这次修改视为哪个节点的输出，LangGraph 据此决定恢复后的执行路径。
⭐
拒绝的本质是"替换 AIMessage 内容"把含有 tool_calls 的 AIMessage 替换成不含 tool_calls 的普通文字消息，tools_condition 就会路由到 END，工具永远不会被执行。
⭐
选择性审核：interrupt_before 是节点级别的要实现"只拦截危险工具"，需把工具按安全性拆进两个不同的 ToolNode，只对危险工具节点设置 interrupt_before。自定义路由函数根据 tool_calls 的工具名决定去哪个节点。

第六课 · Lesson 06

多 Agent 协作：Supervisor 模式

一个"调度 LLM"负责分析任务进度、决定交给哪个专家处理，专家完成后汇报，Supervisor 再决定下一步——直到所有子任务完成。

Supervisor 调度节点 next 字段路由 专家 Agent 设计 多节点循环 vs 单 Agent ReAct
1

为什么需要多 Agent？

第三课的 ReAct Agent 是单个 LLM 自己循环调工具，适合大多数场景。但当任务变得复杂时，单 Agent 方案会遇到瓶颈：

⚠️
工具太多，LLM 容易混乱把几十个工具都绑定到一个 LLM，它往往不知道该选哪个，或者会乱用工具。
⚠️
不同子任务需要不同专业性"查资料"和"写代码"的最佳系统提示完全不同，强行用一个 Agent 处理所有事情效果不好。
⚠️
需要并行或流水线执行复杂任务可以拆成多个子任务，由专门的 Agent 并行或串行处理，提高效率和质量。
对比单 Agent（第三课）多 Agent Supervisor（本课）
工具绑定所有工具绑给同一个 LLM每个专家只绑定自己领域的工具
系统提示一份通用提示每个专家有专属提示，角色更聚焦
决策者LLM 自己决定下一步调哪个工具Supervisor LLM 决定下一步交给哪个专家
适用场景工具数量少、任务相对简单跨领域复合任务、工具数量多
2

Supervisor 模式架构

Supervisor 模式的核心是一个"调度节点"——它不执行任务，只做决策：

┌────────────────────────────────────────┐ │ │ START ──▶ [supervisor] ──▶ [researcher] ──┤ │ │ ├──▶ [analyst] ─────────────┘ │ └──▶ END（当 next == "FINISH"） Supervisor 每次执行： 1. 读取完整的 messages 历史，了解任务进度 2. 判断：还有哪些子任务未完成？ 3. 输出下一个执行者的名字（researcher / analyst / FINISH） 4. 由路由函数把执行流导向对应节点 每个专家执行完后，将结果追加到 messages，再回到 Supervisor 重新评估。
3

next 字段：State 的路由控制器

与前几课相比，State 新增了 next 字段，专门用于 Supervisor 的路由决策：

class State(TypedDict):
    messages: Annotated[list[BaseMessage], add_messages]  # 对话历史（内容）
    next: str   # Supervisor 的路由决策（控制流）："researcher"|"analyst"|"FINISH"
📌
messages 存内容，next 存控制流两个字段职责分离：messages 记录所有专家的工作成果，next 控制下一步执行路径。Supervisor 只写 next，不向 messages 追加内容，避免污染对话历史。
📌
路由函数直接读 next 字段supervisor_router 只需 return state["next"]，逻辑极简。条件边的映射表把字符串值对应到实际节点名。
4

Supervisor 节点设计

MEMBERS = ["researcher", "analyst"]
OPTIONS = MEMBERS + ["FINISH"]

SUPERVISOR_PROMPT = """你是任务协调者，负责把任务分配给合适的专家。
可用专家：
- researcher：查询知识、搜索信息
- analyst   ：数学计算、数据统计

只返回以下选项之一：researcher, analyst, FINISH"""

def supervisor_node(state: State) -> dict:
    messages = [SystemMessage(content=SUPERVISOR_PROMPT)] + state["messages"]
    response = llm.invoke(messages)
    next_agent = response.content.strip()

    # 防御性处理：格式不符时默认结束
    if next_agent not in OPTIONS:
        next_agent = "FINISH"

    # 只更新 next，不向 messages 追加（避免污染历史）
    return {"next": next_agent}

def supervisor_router(state: State) -> str:
    return state["next"]  # 直接返回 next 字段的值
5

专家 Agent 设计

每个专家有独立的 LLM（绑定专属工具）+ 专属系统提示，让 LLM 聚焦在自己的领域：

# 每个专家只绑定自己领域的工具
researcher_llm = llm.bind_tools([search_knowledge, get_latest_info])
analyst_llm    = llm.bind_tools([calculate, get_statistics])

def researcher_node(state: State) -> dict:
    messages = [
        SystemMessage(content="你是知识研究专家，负责查询和整理信息。"),
        *state["messages"],   # 展开完整历史，让专家了解任务背景
    ]
    response = researcher_llm.invoke(messages)
    # 如果有工具调用，执行工具后汇总结论
    if response.tool_calls:
        # ... 执行工具，生成研究结论 ...
        return {"messages": [final_conclusion]}
    return {"messages": [response]}

# analyst_node 结构类似，只是 llm 和系统提示不同

Graph 构建代码

graph_builder = StateGraph(State)

graph_builder.add_node("supervisor", supervisor_node)

graph_builder.add_edge("analyst",    "supervisor")

核心知识点总结

⭐
Supervisor 是"决策节点"，不是"执行节点"它只读取 messages 历史、输出 next 字段，不做任何实际工作。专家节点负责执行，Supervisor 负责调度。
⭐
next 字段实现路由控制State 增加 next: str 专门存放路由决策。Supervisor 写入，router 函数读取，条件边映射到实际节点。messages 存内容，next 存控制流，职责清晰。
⭐
每个专家有独立的 LLM + 工具 + 系统提示researcher 只绑定查询类工具，analyst 只绑定计算类工具。专属系统提示让 LLM 聚焦角色，效果比"全能 Agent"更好。
⭐
Supervisor 模式 vs 单 Agent ReActReAct：agent → tools → agent（LLM 自己决定调哪个工具）；Supervisor：supervisor → specialistA/B → supervisor（专门的调度 LLM 决定交给哪个专家）。复杂任务用 Supervisor，简单任务用 ReAct。
⭐
Supervisor 的智能路由LLM 不需要专门委托就能做的事（如简单计算），执行的专家会顺带处理，Supervisor 识别后直接 FINISH。这是正常的智能行为，让系统更高效。
