第一课 · Lesson 01

Hello Graph —— 最小可运行的 LangGraph

只学 4 个概念，写 2 个节点，先跑起来再说。理解 State、Node、Edge、Graph 的关系，掌握 LangGraph 的基本骨架。

State（TypedDict） Node（节点函数） Edge（有向边） compile + invoke
1

4 个概念，先建立整体印象

LangGraph 的所有程序都由这 4 个东西组成，先认识它们，后面的代码就一目了然：

概念是什么类比
State 一个 TypedDict，整个 Graph 的共享数据包 接力赛的"接力棒"，每个选手都能读写
Node 普通 Python 函数，读 State、返回更新字典 接力赛的"选手"，接棒后做自己的事
Edge 节点之间的有向连线，决定谁先谁后 接力赛的"跑道"，规定跑的顺序
Graph 把以上三者组装起来，compile 后才能运行 整个接力赛的"赛制"
💡 最关键的一点
节点之间不直接传参，全靠读写同一个 State。A 节点把结果写进 State，B 节点从 State 读出来继续处理。
2

State：共享数据包

State 用 TypedDict 定义，声明有哪些字段、分别是什么类型。整个 Graph 运行期间，所有节点都共享这同一个 State 对象。

from typing import TypedDict

class State(TypedDict):
    question: str   # 初始输入
    answer: str     # LLM 写入，后续节点读取
📦
只放"跨节点传递"的数据节点内部的临时变量不需要放进 State。只有"A 写了、B 要读"的数据才放进来。
✏️
节点只返回"改了什么"返回 {"answer": "xxx"} 只更新 answer 字段，其余字段自动保持不变。
3

Node：节点函数

节点就是普通的 Python 函数，签名固定：接收 State，返回 dict。返回的 dict 里只需要包含要更新的字段。

# 节点1：调用 LLM，把回答写入 answer 字段
def llm_node(state: State) -> dict:
    response = llm.invoke(state["question"])
    return {"answer": response.content}   # 只返回要改的字段

# 节点2：打印字数，不修改任何字段
def answer_node(state: State) -> dict:
    print(f"回答字数：{len(state['answer'])}")
    return {}   # 不更新任何字段，返回空字典
⚠️ 必须返回 dict
节点函数必须返回一个 dict，即使什么都不改也要返回 {}。返回 None 会报错。
4

Edge：有向边

Edge 决定节点的执行顺序。用 add_edge(A, B) 表示"A 执行完之后，执行 B"。

START 和 END 是 LangGraph 内置的两个特殊节点，分别代表入口和出口：

from langgraph.graph import StateGraph, START, END

builder.add_edge(START, "llm_node")    # 入口：从 llm_node 开始
builder.add_edge("llm_node", "answer_node")   # llm_node 完成后执行 answer_node
builder.add_edge("answer_node", END)  # 出口：answer_node 完成后结束
✅ add_edge(START, ...) 就是设置入口
add_edge(START, "节点名") 等价于旧写法 set_entry_point("节点名")，用哪个都行，推荐前者更直观。
5

构建与运行

把 State、Node、Edge 组装到 StateGraph 里，然后调用 compile() 锁定配置，最后用 invoke() 传入初始数据运行：

# 1. 创建 builder，传入 State 类型
builder = StateGraph(State)

# 2. 注册节点
builder.add_node("llm_node", llm_node)
builder.add_node("answer_node", answer_node)

# 3. 连接边
builder.add_edge(START, "llm_node")
builder.add_edge("llm_node", "answer_node")
builder.add_edge("answer_node", END)

# 4. 编译（锁定配置，生成可执行对象）
graph = builder.compile()

# 5. 运行（传入初始 State，只需填有值的字段）
result = graph.invoke({"question": "用一句话介绍 LangGraph"})
print(result["answer"])
💡 invoke 的参数是初始 State
invoke({"question": "..."}) 只需要传有初始值的字段，answer 字段没有初始值可以不传，LangGraph 会自动补空。
6

完整代码（01_hello_graph.py）

import os
from typing import TypedDict

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END

load_dotenv()

llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("MODEL"),
)

# State：整个 Graph 的唯一数据载体
class State(TypedDict):
    question: str
    answer: str

# 节点1：调用 LLM，写入 answer
def llm_node(state: State) -> dict:
    response = llm.invoke(state["question"])
    return {"answer": response.content}

# 节点2：打印字数，不改任何字段
def answer_node(state: State) -> dict:
    print(f"回答字数：{len(state['answer'])}")
    return {}

# 构建 Graph
builder = StateGraph(State)
builder.add_node("llm_node", llm_node)
builder.add_node("answer_node", answer_node)
builder.add_edge(START, "llm_node")
builder.add_edge("llm_node", "answer_node")
builder.add_edge("answer_node", END)

graph = builder.compile()

# 运行
if __name__ == "__main__":
    result = graph.invoke({"question": "用一句话介绍 LangGraph"})
    print("问题：", result["question"])
    print("回答：", result["answer"])

知识点总结

🗂️
State 是唯一的数据通道 节点之间不直接传参，全靠读写同一个 State。A 写入，B 读取。
↩️
节点只返回"改了什么" return {"answer": "xxx"} 只更新 answer，其余字段保持不变。return {} 表示什么都不改。
🔗
add_edge(START, ...) 设置入口 START 是内置入口标记，END 是内置出口标记，都从 langgraph.graph 导入。
⚙️
compile() 之后才能运行 StateGraph 是 builder，compile() 才生成真正可执行的 graph 对象。
▶️
invoke() 传入初始 State 只需填有初始值的字段。invoke({"question": "..."}) 即可，未填字段 LangGraph 自动补空。

踩坑记录

🚫 坑1：用了 Graph 而不是 StateGraph

LangChain 里有个 Graph 类（langchain_core.runnables.graph），但 LangGraph 用的是 StateGraph。两个名字很像，但完全是不同的东西。

# ❌ 错误：这是 LangChain 内部类，不是 LangGraph 的
from langchain_core.runnables.graph import Graph

# ✅ 正确：从 langgraph.graph 导入
from langgraph.graph import StateGraph
🚫 坑2：忘记调用 compile()

build 完之后直接 invoke() 会报错，必须先 compile()。

# ❌ 错误：builder 不是可执行对象
builder.invoke({...})

# ✅ 正确：compile 后的对象才能 invoke
graph = builder.compile()
graph.invoke({...})
🚫 坑3：节点返回 None

节点函数必须返回 dict。忘记写 return 或者 return None 会导致 State 更新失败或报错。即使什么都不改，也要写 return {}。

# ❌ 错误：没有返回值
def my_node(state):
    print(state["answer"])   # 忘记 return

# ✅ 正确：不改字段也要返回空字典
def my_node(state):
    print(state["answer"])
    return {}
🚫 坑4：START 和 END 的导入路径

START 和 END 都在 langgraph.graph 里，不要从 langgraph.constants 单独导入（旧版写法，新版推荐统一从 graph 导入）。

# ✅ 推荐写法（一行搞定）
from langgraph.graph import StateGraph, START, END
🚫 坑5：llm.invoke() 传字符串 vs 传消息列表

llm.invoke(string) 和 llm.invoke([HumanMessage(...)]) 都能用，效果基本一样。第一课用字符串更简洁，从第二课开始引入 HumanMessage。

# 两种写法都正确
response = llm.invoke("你好")                            # 字符串
response = llm.invoke([HumanMessage(content="你好")])   # 消息列表
第二课 · Lesson 02

消息格式 —— HumanMessage 与 SystemMessage

Graph 结构完全不变，只学一件事：如何用消息列表调用 LLM，以及如何用 SystemMessage 给 LLM 设定角色、改变回答风格。

HumanMessage SystemMessage 消息列表格式 State 新增字段
1

为什么要用消息列表，而不是字符串？

第一课用 llm.invoke("字符串")，这是最简单的写法。但实际开发中几乎总是用消息列表，原因只有一个：

💡 字符串只能传问题，消息列表可以传"角色设定"
用字符串，LLM 按默认风格回答。用消息列表，你可以用 SystemMessage 告诉 LLM "你是一个什么样的助手"，它就会按你设定的风格来回答。
写法能做什么适合场景
llm.invoke("问题") 传入用户问题，LLM 按默认风格回答 第一课学语法，快速验证
llm.invoke([HumanMessage(...)]) 同上，但用了标准消息格式 实际开发的基础写法
llm.invoke([SystemMessage(...), HumanMessage(...)]) 设定角色 + 传入问题，LLM 按角色风格回答 几乎所有实际应用
2

三种消息类型

LangChain 定义了几种消息类型，最常用的三种：

from langchain_core.messages import HumanMessage, SystemMessage, AIMessage
👤
HumanMessage —— 用户说的话 就是用户这一轮输入的内容。每次提问都用它包装。
HumanMessage(content="什么是人工智能？")
⚙️
SystemMessage —— 给 LLM 的角色设定 放在列表最前面，告诉 LLM "你是谁、你要怎么回答"。
不是用户说的话，是给 LLM 的"幕后指令"。
SystemMessage(content="你是一个幽默的助手")
🤖
AIMessage —— LLM 的回复 llm.invoke() 返回的就是 AIMessage 对象。
用 .content 取出文本内容。本课暂时不需要手动创建它。
3

SystemMessage：同一个问题，不同风格的回答

同一个问题 "什么是人工智能？"，只改 SystemMessage，LLM 的回答风格就完全不同：

# 正经学术风格
llm.invoke([
    SystemMessage(content="你是严谨的学术助手，用正式专业的语言回答，控制在3句话以内"),
    HumanMessage(content="什么是人工智能？"),
])

# 幽默风格
llm.invoke([
    SystemMessage(content="你是幽默风趣的助手，用轻松有趣的语气回答，可以加比喻，控制在3句话以内"),
    HumanMessage(content="什么是人工智能？"),
])
✅ SystemMessage 就是 LLM 的"人设"
你写给 LLM 的 prompt 越具体，它的表现越符合预期。"你是一个严谨的助手，回答控制在3句话以内"比"你是个助手"效果好得多。
4

State 新增 role 字段

本课在 State 里加了一个 role 字段，用来存放 SystemMessage 的内容。这样同一个 Graph，传入不同的 role，就能得到不同风格的回答：

class State(TypedDict):
    question: str
    role: str      # 新增：SystemMessage 的内容，由 invoke 时传入
    answer: str

节点函数从 State 里读取 role 来构建 SystemMessage：

def llm_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content=state["role"]),      # 从 State 读角色设定
        HumanMessage(content=state["question"]),   # 从 State 读问题
    ])
    return {"answer": response.content}
⚠️ role 字段不是必须的
role 字段是这节课为了演示"State 可以控制节点行为"而加的。如果你的角色是固定的，直接把 SystemMessage 的内容写死在节点函数里也完全可以。
5

完整代码（02_messages.py）

import os
from typing import TypedDict

from dotenv import load_dotenv
from langchain_core.messages import HumanMessage, SystemMessage
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END

load_dotenv()

llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("MODEL"),
)

class State(TypedDict):
    question: str
    role: str      # LLM 的角色设定
    answer: str

def llm_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content=state["role"]),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def answer_node(state: State) -> dict:
    print(f"回答字数：{len(state['answer'])}")
    return {}

builder = StateGraph(State)
builder.add_node("llm_node", llm_node)
builder.add_node("answer_node", answer_node)
builder.add_edge(START, "llm_node")
builder.add_edge("llm_node", "answer_node")
builder.add_edge("answer_node", END)

graph = builder.compile()

if __name__ == "__main__":
    question = "什么是人工智能？"

    print("=" * 50)
    print("角色1：正经的学术助手")
    print("=" * 50)
    result1 = graph.invoke({
        "question": question,
        "role": "你是一个严谨的学术助手，用正式、专业的语言回答问题，回答控制在3句话以内。",
    })
    print(result1["answer"])

    print("\n" + "=" * 50)
    print("角色2：幽默的助手")
    print("=" * 50)
    result2 = graph.invoke({
        "question": question,
        "role": "你是一个幽默风趣的助手，用轻松有趣的语气回答，可以加比喻，回答控制在3句话以内。",
    })
    print(result2["answer"])

知识点总结

📨
消息列表是 LLM 的标准输入格式 llm.invoke([消息1, 消息2, ...])。字符串写法是简写，内部自动转成 HumanMessage。
⚙️

📦

踩坑记录

🚫 坑1：忘记取 .content，直接打印 response

llm.invoke() 返回的是 AIMessage 对象，不是字符串。直接 print 会看到 content='...' response_metadata={...} 这样的对象格式。

# ❌ 打印出对象，不是纯文本
response = llm.invoke([...])
print(response)       # content='AI是...' response_metadata={...}

# ✅ 取 .content 才是文本
print(response.content)   # AI是...
🚫 坑2：SystemMessage 和 HumanMessage 顺序写反

消息列表的顺序有意义。SystemMessage 必须放在最前面，HumanMessage 放后面。如果反过来写，很多模型会忽略 SystemMessage 或表现异常。

# ❌ 顺序写反，SystemMessage 不一定生效
llm.invoke([HumanMessage(...), SystemMessage(...)])

# ✅ SystemMessage 在前
llm.invoke([SystemMessage(...), HumanMessage(...)])
🚫 坑3：invoke 时没传 role 字段，导致 KeyError

State 里定义了 role 字段，invoke 时如果没传，节点访问 state["role"] 会报 KeyError。TypedDict 只是类型提示，不提供默认值。

# ❌ 缺少 role 字段，节点里 state["role"] 报 KeyError
graph.invoke({"question": "..."})

# ✅ 必须传入 State 里所有节点会读取的字段
graph.invoke({"question": "...", "role": "你是..."})
第三课 · Lesson 03

节点链 —— 数据在节点间流动

Graph 还是线性的，只加一个节点。重点是理解：前面节点写入 State，后面节点就能读到——这就是"数据流水线"。节点也不是只能调 LLM，普通 Python 逻辑同样可以是一个节点。

3 节点串联 字段依次写入 节点可以是纯 Python State 累积字段
1

流水线：前面写，后面读

def llm_node(state: State) -> dict:

def count_node(state: State) -> dict:
    count = len(state["answer"])   # 读取 llm_node 写入的字段

def print_node(state: State) -> dict:

    print(f"回答：{state['answer']}")
    print(f"字数：{state['word_count']}")  # 读取 count_node 写入的字段
    return {}
⚠️ count_node 没有调 LLM
节点就是普通 Python 函数，不是每个节点都要调 LLM。count_node 只做了 len()，同样是一个合法的节点。以后你会遇到做格式转换、过滤、数据库查询的节点，都是同样的写法。

完整代码（03_pipeline.py）

import os
from typing import TypedDict

from dotenv import load_dotenv
from langchain_core.messages import HumanMessage, SystemMessage
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END

load_dotenv()

llm = ChatOpenAI(
    api_key=os.getenv("API_KEY"),
    base_url=os.getenv("BASE_URL"),
    model=os.getenv("MODEL"),
)

class State(TypedDict):
    question:   str
    role:       str
    answer:     str
    word_count: int

def llm_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content=state["role"]),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def count_node(state: State) -> dict:
    return {"word_count": len(state["answer"])}

def print_node(state: State) -> dict:
    print(f"问题：{state['question']}")
    print(f"回答：{state['answer']}")
    print(f"字数：{state['word_count']}")
    return {}

builder = StateGraph(State)
builder.add_node("llm_node",   llm_node)
builder.add_node("count_node", count_node)
builder.add_node("print_node", print_node)

builder.add_edge(START,        "llm_node")
builder.add_edge("llm_node",   "count_node")
builder.add_edge("count_node", "print_node")
builder.add_edge("print_node", END)

graph = builder.compile()

if __name__ == "__main__":
    result = graph.invoke({
        "question": "用两句话介绍一下 Python 语言",
        "role":     "你是简洁的技术助手，回答控制在两句话以内。",
    })
    print("\n最终 State：", result)

🚫 坑1：count_node 里读不到 answer

如果 llm_node 和 count_node 之间没有 add_edge，或者顺序写反了，count_node 执行时 answer 还是空字符串。

# ❌ 顺序写反：count_node 先执行，answer 还没被写入

builder.add_edge("count_node",   "llm_node")

# ✅ 正确：llm_node 先写 answer，count_node 再读
builder.add_edge(START,        "llm_node")
builder.add_edge("llm_node",   "count_node")
🚫 坑2：State 定义了字段但没有在 invoke 时传，也没有节点写入

State 里定义了 word_count: int，但如果 count_node 没有正确返回，invoke 返回的 State 里 word_count 会是 None 而不是 0。TypedDict 只是类型提示，不提供默认值。

    "question":   "...",
    "role":       "...",
    "word_count": 0,   # 可选：给个初始值更保险
})
🚫 坑3：add_node 的顺序不影响执行顺序

执行顺序由 add_edge 决定，不是 add_node 的顺序。即使你先 add_node count_node 再 add_node llm_node，只要 edge 连对了，执行顺序就是对的。

# 这样注册节点，执行顺序依然是 llm → count → print
builder.add_node("print_node", print_node)   # 先注册
builder.add_node("count_node", count_node)
builder.add_node("llm_node",   llm_node)    # 后注册
# 执行顺序由下面的 add_edge 决定，与 add_node 顺序无关
第四课 · Lesson 04

条件路由 —— 根据 State 走不同的路

前三课的 Graph 都是固定线性的。本课第一次让图"分叉"——根据 State 的值决定走哪条路。只学一个新 API：add_conditional_edges。

add_conditional_edges 路由函数 映射字典 动态路径
1

add_edge vs add_conditional_edges

前三课只用了 add_edge，路径是固定的。add_conditional_edges 让路径变成"看情况"：

add_edgeadd_conditional_edges
路径 固定：A 完成后永远去 B 动态：A 完成后根据 State 决定去哪
参数 (源节点, 目标节点) (源节点, 路由函数, 映射字典)
使用场景 顺序固定的步骤 需要根据条件走不同分支 
# add_edge：固定路径
builder.add_edge("node_a", "node_b")    # A 完成后永远去 B

# add_conditional_edges：动态路径
builder.add_conditional_edges(
    "node_a",       # 从哪个节点出发
    router_fn,       # 路由函数：读 State，返回下一节点的名字
    {               # 映射字典：路由函数返回值 → 实际节点名
        "node_b": "node_b",
        "node_c": "node_c",
    }
)
2

路由函数：读 State，返回节点名

路由函数的写法固定：接收 State，返回一个字符串（下一个节点的名字）。它只做"决策"，不修改 State：

def route_by_mode(state: State) -> str:
    if state["mode"] == "brief":
        return "brief_node"     # 返回节点名字符串
    return "detailed_node"       # 返回另一个节点名
💡 路由函数只做决策，不做处理
路由函数只读 State 然后返回字符串，不调 LLM，不写 State。真正的业务逻辑放在各自的节点函数里，两者职责分离。
3

映射字典：为什么看起来"多余"

{"brief_node": "brief_node"} 看起来左右一样，像是废话。它的真正作用是：允许路由函数的返回值与节点名不同：

# 路由函数返回 "short"，但实际节点名是 "brief_node"
def router(state) -> str:
    return "short" if state["mode"] == "brief" else "long"

builder.add_conditional_edges(
    START, router,
    {
        "short": "brief_node",     # "short" → 实际节点 "brief_node"
        "long":  "detailed_node",  # "long"  → 实际节点 "detailed_node"
    }
)
✅ 初学阶段保持左右一样就行
路由函数直接返回节点名，映射字典写成 {"brief_node": "brief_node"}，最简单清晰。等熟悉了再考虑分离返回值和节点名。
4

完整代码（04_conditional.py）

import os
from typing import TypedDict

from dotenv import load_dotenv
from langchain_core.messages import HumanMessage, SystemMessage
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END

load_dotenv()
llm = ChatOpenAI(api_key=os.getenv("API_KEY"), base_url=os.getenv("BASE_URL"), model=os.getenv("MODEL"))

class State(TypedDict):
    question: str
    mode:     str   # "brief" 或 "detailed"
    answer:   str

# 路由函数：读 State，返回节点名
def route_by_mode(state: State) -> str:
    if state["mode"] == "brief":
        return "brief_node"
    return "detailed_node"

def brief_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content="你是简洁的助手，用1-2句话回答，不展开细节。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def detailed_node(state: State) -> dict:
    response = llm.invoke([
        SystemMessage(content="你是耐心的助手，详细解释，可以分点说明，越完整越好。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

builder = StateGraph(State)
builder.add_node("brief_node",    brief_node)
builder.add_node("detailed_node", detailed_node)

# 条件边：从 START 出发，由 route_by_mode 决定去哪
builder.add_conditional_edges(
    START,
    route_by_mode,
    {
        "brief_node":    "brief_node",
        "detailed_node": "detailed_node",
    }
)
builder.add_edge("brief_node",    END)
builder.add_edge("detailed_node", END)

graph = builder.compile()

if __name__ == "__main__":
    q = "什么是递归？"

    result1 = graph.invoke({"question": q, "mode": "brief"})
    print("【简洁】", result1["answer"])

    result2 = graph.invoke({"question": q, "mode": "detailed"})
    print("【详细】", result2["answer"])

踩坑记录

🚫 坑1：路由函数返回的字符串不在映射字典里

路由函数返回了 "brief"，但映射字典里只有 "brief_node"，LangGraph 找不到对应节点，会报错。路由函数的返回值必须和映射字典的 key 完全一致。

# ❌ 返回 "brief"，但字典 key 是 "brief_node"
def router(state): return "brief"
builder.add_conditional_edges(START, router, {"brief_node": "brief_node"})

# ✅ 返回值与 key 一致
def router(state): return "brief_node"
builder.add_conditional_edges(START, router, {"brief_node": "brief_node"})
🚫 坑2：分支节点忘记连到 END

两个分支节点都要有通向 END 的边。漏掉其中一个，那条路走完就"卡住"了，Graph 不知道该结束还是继续。

# ❌ 只连了 brief_node，detailed_node 走完没有出口
builder.add_edge("brief_node", END)

# ✅ 两个分支都要连 END
builder.add_edge("brief_node",    END)
builder.add_edge("detailed_node", END)
🚫 坑3：invoke 时没传 mode 字段，路由函数报 KeyError

路由函数里访问 state["mode"]，但 invoke 时没有传 mode，就会 KeyError。解决方法：invoke 时传 mode，或者路由函数里用 state.get("mode", "brief") 设默认值。

# ✅ 路由函数加上 .get() 防止 KeyError
def route_by_mode(state: State) -> str:
    mode = state.get("mode", "brief")   # 没传 mode 时默认 "brief"
    return "brief_node" if mode == "brief" else "detailed_node"
第五课 · Lesson 05

多路汇聚 —— Fan-in

第四课两条分支各自走向 END。本课把分支数加到 3，并让三条路都汇入同一个 print_node，再统一结束。只学一件事：多个节点可以都连向同一个节点。

3条分支 fan-in 汇聚 公共出口节点
1

Fan-in：多条路汇入同一个节点

第四课的结构：每条分支各自走向 END。

第五课的结构：每条分支走向同一个 print_node，再由 print_node 统一走向 END。

第四课第五课
分支数23
分支后各自 → END全部 → print_node → END
新概念add_conditional_edgesfan-in（多路汇聚）
2

为什么需要汇聚节点？

💡 避免在每个分支里重复写相同的逻辑
如果没有 fan-in，打印逻辑要在 brief / detailed / bullet 三个节点里各写一遍。
把公共逻辑提取成 print_node，所有分支都汇进来，写一次就够了。 
# ❌ 没有 fan-in：每个节点都要自己打印
def brief_node(state):
    ...
    print(state["answer"])   # 重复
    return {"answer": ...}

def detailed_node(state):
    ...
    print(state["answer"])   # 重复
    return {"answer": ...}

# ✅ 有 fan-in：打印逻辑只写一次
def print_node(state):
    print(state["answer"])   # 只在这里写一次
    return {}
3

三条分支汇入同一个节点

用法很简单：对每个分支节点各写一条 add_edge(..., "print_node")：

# 三条分支都连向 print_node
builder.add_edge("brief",    "print_node")
builder.add_edge("detailed", "print_node")
builder.add_edge("bullet",   "print_node")

# print_node 只需要一条出边
builder.add_edge("print_node", END)

完整代码（05_fanin.py）

class State(TypedDict):
    question: str
    mode: str    # "brief" | "detailed" | "bullet"
    answer: str

def route_by_mode(state: State) -> str:
    mode = state.get("mode", "brief")
    if mode == "brief":    return "brief"
    elif mode == "detailed": return "detailed"
    return "bullet"

def brief_node(state):
    response = llm.invoke([
        SystemMessage(content="你是简洁的助手，用1-2句话回答。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def detailed_node(state):
    response = llm.invoke([
        SystemMessage(content="你是耐心的助手，详细分点说明。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def bullet_node(state):
    response = llm.invoke([
        SystemMessage(content="用5条以内的要点回答，每条以 - 开头。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answer": response.content}

def print_node(state):             # 汇聚节点
    print(f"\n【模式：{state['mode']}】")
    print(state["answer"])
    return {}

builder = StateGraph(State)
builder.add_node("brief",      brief_node)
builder.add_node("detailed",   detailed_node)
builder.add_node("bullet",     bullet_node)
builder.add_node("print_node", print_node)

builder.add_conditional_edges(
    START, route_by_mode,
    {"brief": "brief", "detailed": "detailed", "bullet": "bullet"}
)

builder.add_edge("brief",    "print_node")  # fan-in
builder.add_edge("detailed", "print_node")  # fan-in
builder.add_edge("bullet",   "print_node")  # fan-in
builder.add_edge("print_node", END)

graph = builder.compile()

第六课 · Lesson 06

消息列表 —— 追加而不是覆盖

前五课的 State 字段每次都是覆盖旧值。本课引入消息列表：节点返回的新消息会追加到列表末尾，旧消息不会丢失。这是后面多轮对话和 ReAct Agent 的基础。

add_messages reducer Annotated 类型标注 追加 vs 覆盖 BaseMessage 列表
1

覆盖 vs 追加

前五课的字段默认是覆盖行为，消息列表需要追加行为：

普通字段（覆盖）消息列表（追加）
字段类型 answer: str messages: Annotated[list[BaseMessage], add_messages]
节点返回 {"answer": "新值"} {"messages": [新消息]}
结果 旧值被替换 新消息追加到列表末尾
旧内容 丢失 保留
💡 为什么对话历史必须追加？
LLM 需要看到完整的对话记录才能理解上下文。如果每次覆盖，LLM 就只能看到最新一条消息，不知道之前说了什么。
2

Annotated + add_messages：告诉 LangGraph 用追加而不是覆盖

Annotated 是 Python 的类型标注工具，可以给类型附加额外信息。LangGraph 用第二个参数来决定这个字段怎么合并：

from typing import Annotated
from langgraph.graph.message import add_messages

# 普通字段：新值覆盖旧值（默认行为）
answer: str

# 消息列表：用 add_messages 函数合并，效果是追加
messages: Annotated[list[BaseMessage], add_messages]
#          ↑ 类型         ↑ reducer 函数（告诉 LangGraph 怎么合并）
✅ add_messages 做了什么
当节点返回 {"messages": [新消息]} 时，LangGraph 调用 add_messages(旧列表, [新消息])，把两个列表拼在一起。你不需要自己管理列表，它自动追加。

from typing import Annotated, TypedDict

from dotenv import load_dotenv
from langchain_core.messages import BaseMessage, HumanMessage
from langchain_openai import ChatOpenAI
from langgraph.graph import StateGraph, START, END
from langgraph.graph.message import add_messages

llm = ChatOpenAI(api_key=os.getenv("API_KEY"), base_url=os.getenv("BASE_URL"), model=os.getenv("MODEL"))

    messages: Annotated[list[BaseMessage], add_messages]

    return {"messages": [response]}

def print_node(state: State) -> dict:
    print(f"\n对话共 {len(state['messages'])} 条消息：")
    for msg in state["messages"]:
        role = msg.__class__.__name__.replace("Message", "")
        print(f"  [{role}] {msg.content[:50]}")
    return {}

builder.add_node("llm_node",   llm_node)
builder.add_node("print_node", print_node)
builder.add_edge(START,        "llm_node")
builder.add_edge("llm_node",   "print_node")
builder.add_edge("print_node", END)

第七课 · Lesson 07

工具定义 —— @tool 与 bind_tools

告诉 LLM 有哪些函数可以调用。LLM 会根据问题自己决定：直接回答，还是声明要调用某个工具。本课只观察 LLM 的决策，工具的实际执行留到下一课。

@tool 装饰器 bind_tools() tool_calls 两种回复形态
1

工具是什么

前六课的 LLM 只能回答文字。加了工具之后，LLM 可以"声明"自己想调用某个函数来获取信息或执行操作。

没有工具有工具
LLM 能做什么只能回答文字可以声明调用函数
回复形式response.content（文字）response.content 或 response.tool_calls
适合场景知识问答计算、查询、操作等需要执行代码的场景
💡 LLM 不会真的"执行"工具
LLM 只是在回复里声明"我要调用 add，参数是 a=25, b=37"。真正执行 add(25, 37) 是你的代码的事。本课先看声明，下一课再看执行。
2

@tool 装饰器

在普通函数上加 @tool，它就变成了 LLM 可以调用的工具：

from langchain_core.tools import tool

@tool
def add(a: int, b: int) -> int:
    """计算两个整数的和"""   # ← LLM 靠这个描述判断要不要用这个工具
    return a + b
📝
docstring 是给 LLM 看的 LLM 读 docstring 来判断这个工具能做什么、什么时候该用它。描述越准确，LLM 的判断越准。
🏷️
参数类型注解是必须的 a: int, b: int 告诉 LLM 参数的类型，LLM 据此填写正确的参数值。没有类型注解可能报错。
3

bind_tools()：把工具注册给 LLM

tools = [add, multiply, get_weather]

# 普通 LLM：只能回答文字
llm

# 绑定工具后的 LLM：知道有工具可用，可以声明调用
llm_with_tools = llm.bind_tools(tools)
✅ bind_tools 不改变 LLM 本身
bind_tools() 返回一个新的 LLM 对象，原来的 llm 不变。用 llm_with_tools 调用时 LLM 知道有工具，用普通 llm 调用时不知道。
4

两种回复形态

llm_with_tools.invoke() 返回的 AIMessage 有两种形态：

# 问题不需要工具 → 直接回答
response.content    = "天空是蓝色的原因是..."
response.tool_calls = []   # 空列表

# 问题需要工具 → 声明调用
response.content    = ""   # 空字符串
response.tool_calls = [
    {
        "name": "add",
        "args": {"a": 25, "b": 37},
        "id":   "call_xxx",
    }
]

判断 LLM 的决策只需要检查 tool_calls 是否为空：

if response.tool_calls:
    print("LLM 要调用工具", response.tool_calls)
else:
    print("LLM 直接回答", response.content)

from langchain_core.tools import tool

@tool
def add(a: int, b: int) -> int:
    """计算两个整数的和"""
    return a + b

@tool
def multiply(a: int, b: int) -> int:
    """计算两个整数的乘积"""
    return a * b

@tool
def get_weather(city: str) -> str:
    """查询指定城市的天气"""
    data = {"北京": "晴天 25°C", "上海": "多云 22°C"}
    return data.get(city, f"暂无 {city} 的天气数据")

    messages: Annotated[list[BaseMessage], add_messages]

def llm_node(state: State) -> dict:
    response = llm_with_tools.invoke(state["messages"])
    return {"messages": [response]}

def print_node(state: State) -> dict:
    last = state["messages"][-1]
    if last.tool_calls:
        print("LLM 决定调用工具：")
        for tc in last.tool_calls:
            print(f"  工具名：{tc['name']}，参数：{tc['args']}")
    else:
        print(f"LLM 直接回答：{last.content}")
    return {}

ReAct Agent：工具真正被执行
ToolNode + tools_condition + 循环结构
ReAct 是什么

第七课让 LLM 声明要调用哪个工具，但工具并没有真正运行。
第八课补上最后一步：自动执行工具，把结果喂回 LLM，LLM 再决定下一步，循环直到不需要工具为止。

这个模式叫 ReAct（Reason + Act + Observe）：

步骤 含义 对应节点
Reason LLM 思考要做什么，要不要调用工具 agent 节点
Act 执行工具，得到结果 tools 节点（ToolNode）
Observe 把工具结果追加到消息列表，LLM 观察后再推理 再次进入 agent
与第七课的唯一区别： 第七课 print_node 只是"看"tool_calls，第八课 ToolNode 真正"运行"tool_calls。
新概念 1：ToolNode

ToolNode 是 LangGraph 内置节点，做三件事：

  1. 读取最后一条 AIMessage 的 tool_calls
  2. 找到对应的工具函数，用给定参数执行
  3. 把结果包装成 ToolMessage 追加到 messages
from langgraph.prebuilt import ToolNode

tools = [add, multiply]
# 传入工具列表即可，不需要手写执行逻辑
builder.add_node("tools", ToolNode(tools))

执行后 messages 列表新增一条 ToolMessage：

ToolMessage(
    content="42",           # 工具的返回值（字符串化）
    tool_call_id="xxx",  # 对应 AIMessage 里的 tool_call id
    name="multiply"       # 工具名称
)
新概念 2：tools_condition

tools_condition 是 LangGraph 内置路由函数，逻辑非常简单：

# 内置实现等价于这个逻辑：
def tools_condition(state):
    last = state["messages"][-1]
    if last.tool_calls:
        return "tools"   # 有工具调用 → 去执行工具
    return END          # 没有工具调用 → 结束

用法：

from langgraph.prebuilt import tools_condition

builder.add_conditional_edges("agent", tools_condition)
不需要传第三个参数（映射字典）——tools_condition 返回的字符串 "tools" 刚好对应节点名，END 也是 LangGraph 识别的特殊值，所以默认映射就够用。
新概念 3：循环结构

前七课的 Graph 都是单向的（从 START 到 END 一条路）。
第八课第一次出现循环：tools → agent → tools → agent → ...

builder.add_edge(START,    "agent")
builder.add_conditional_edges("agent", tools_condition)
builder.add_edge("tools",  "agent")  # ← 关键：工具执行后回到 agent

循环终止条件：LLM 不再声明 tool_calls（直接给出最终回答），tools_condition 返回 END。

对话记忆：MemorySaver
让两次 invoke() 之间保留上下文
为什么需要记忆

前八课的所有 invoke() 都是无状态的——每次调用都从空 State 开始，上一次的内容完全消失：

# 第一次
graph.invoke({"messages": [HumanMessage("我叫小明")]})
# LLM："你好小明！"

# 第二次（State 清空了）
graph.invoke({"messages": [HumanMessage("我叫什么名字？")]})
# LLM："抱歉，我不知道你的名字。"  ← 完全忘了

真实的聊天机器人需要记住对话历史。MemorySaver 解决这个问题：它在两次 invoke() 之间把 State 持久化，下次恢复。

新概念：MemorySaver

MemorySaver 是一个 checkpointer（检查点存储器），传给 compile()：

from langgraph.checkpoint.memory import MemorySaver

memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

加了 checkpointer 之后，每次 invoke() 结束时 LangGraph 会自动把当前 State 保存；下次同一个对话线程进来，先把 State 恢复出来，再追加新消息。

Graph 的节点代码不需要任何改动——MemorySaver 完全在 Graph 外部工作，对节点透明。
thread_id：区分不同对话

加了 checkpointer 之后，invoke() 必须带上 config 参数指定是哪个对话线程：

config = {"configurable": {"thread_id": "对话A"}}
graph.invoke(input, config=config)
情况 结果
两次 invoke() 用相同 thread_id State 叠加，LLM 看到完整历史
两次 invoke() 用不同 thread_id 独立的对话，互不干扰
不传 config（或没有 checkpointer） 每次从空 State 开始，无记忆

from langgraph.graph.message import add_messages

    messages: Annotated[list[BaseMessage], add_messages]

def llm_node(state: State) -> dict:

    return {"messages": [response]}

builder = StateGraph(State)
builder.add_node("llm_node", llm_node)
builder.add_edge(START, "llm_node")
builder.add_edge("llm_node", END)

# ★ 唯一变化：compile() 传入 checkpointer
memory = MemorySaver()
graph = builder.compile(checkpointer=memory)

# 每次 invoke() 带上 config
config_a = {"configurable": {"thread_id": "对话A"}}

# 第1轮
graph.invoke(
    {"messages": [HumanMessage(content="你好，我叫小明")]},
    config=config_a,
)

# 第2轮（同一 thread_id → LLM 记得小明）
result = graph.invoke(
    {"messages": [HumanMessage(content="你还记得我叫什么名字吗？")]},
    config=config_a,
)
print(result["messages"][-1].content)
# → "你叫小明！"

# 第3轮（不同 thread_id → 全新对话）
config_b = {"configurable": {"thread_id": "对话B"}}
result = graph.invoke(
    {"messages": [HumanMessage(content="你还记得我叫什么名字吗？")]},
    config=config_b,
)
print(result["messages"][-1].content)
# → "抱歉，你还没有告诉我你的名字。"

  llm_node 运行，LLM 回复
  追加：AIMessage("你好小明！")

第2轮 invoke(thread_id="对话A")
  MemorySaver 恢复：messages = [Human("你好，我叫小明"), AI("你好小明！")]

  llm_node 运行，LLM 看到完整历史，知道叫小明

知识点总结
概念 说明
MemorySaver() 内存 checkpointer，存在 Python 进程内，程序重启后丢失
compile(checkpointer=...) 启用持久化，每次 invoke() 后自动保存 State
config={"configurable": {"thread_id": ...}} 指定对话线程，相同 id 共享历史，不同 id 互相隔离
节点代码完全不用改： MemorySaver 在 Graph 执行框架层工作， llm_node 看到的 state["messages"] 已经包含了之前轮次的历史，无感知。
踩坑记录
坑 1：加了 checkpointer 但忘传 config

compile 时传了 checkpointer=memory，但 invoke 时没传 config，LangGraph 会报错：
ValueError: checkpointer requires configurable "thread_id"
每次 invoke() 都必须带 config={"configurable": {"thread_id": "..."}}。

坑 2：MemorySaver 重启后丢失

MemorySaver 把数据存在 Python 对象里，程序重启后全部消失。
需要持久化可换成 SqliteSaver（存文件）或 PostgresSaver（存数据库），用法相同，只是初始化方式不同。

坑 3：消息列表会越来越长

每轮对话追加 2 条消息（Human + AI），对话越长 messages 越大，LLM 的 token 消耗越多。
生产环境通常需要截断历史（只保留最近 N 条），这是更高级的话题。

10
流式输出：graph.stream()
节点完成就拿到结果，不等 Graph 全部跑完
invoke() vs stream()

前九课都用 invoke()：等 Graph 里所有节点都跑完，才一次性返回最终 State。

stream() 改变了这个行为：每个节点执行完就立刻 yield 一次，调用方可以边等边处理。

方法 返回值 何时返回
invoke() 最终 State（dict） 整个 Graph 跑完后
stream() 迭代器，每次 yield 一个 chunk 每个节点完成后 yield 一次
Graph 的节点、State、compile() 完全不用改——stream() 只是换了一种调用方式。
stream_mode="updates"：拿到每个节点的输出 delta

每个 chunk 是一个 dict：{节点名: 该节点 return 的内容}

for chunk in graph.stream(input, stream_mode="updates"):
    node_name   = list(chunk.keys())[0]   # 是哪个节点
    node_output = chunk[node_name]         # 该节点返回的 dict
    print(node_name, node_output)

对于本课的两节点 Graph（llm_node → print_node），输出是：

llm_node    {"messages": [AIMessage("Python 是...")]}
print_node  {}    ← print_node 返回空 dict
stream_mode="values"：每步后的完整 State 快照

每个 chunk 是当前完整的 State，包含所有字段的最新值。

for i, snapshot in enumerate(graph.stream(input, stream_mode="values")):
    print(i, len(snapshot["messages"]), "条消息")
0  1条消息   ← 初始输入（只有 HumanMessage）
1  2条消息   ← llm_node 完成后（追加了 AIMessage）
2  2条消息   ← print_node 完成后（没改 State，条数不变）
stream_mode chunk 内容 适合场景
"updates" 只有本节点改动的字段 调试、监控每个节点做了什么
"values" 完整 State 快照 前端需要展示当前完整状态
完整代码（10_streaming.py）
# State、节点、Graph 和第六课完全一样，只有调用方式变了
input_state = {"messages": [HumanMessage(content="用3句话介绍Python")]}

# invoke()：一次性返回
result = graph.invoke(input_state)
print(result["messages"][-1].content)

# stream() updates：每节点完成 yield 一次
for chunk in graph.stream(input_state, stream_mode="updates"):
    node_name = list(chunk.keys())[0]
    print(f"节点 [{node_name}] 完成：{chunk[node_name]}")

# stream() values：每步后完整 State
for i, snapshot in enumerate(graph.stream(input_state, stream_mode="values")):
    print(f"快照 {i}：messages 共 {len(snapshot['messages'])} 条")
知识点总结
概念 说明
graph.stream(input) 返回迭代器，每个节点完成后 yield 一个 chunk
stream_mode="updates" chunk = {节点名: 该节点返回的 dict}，只有变化的字段
stream_mode="values" chunk = 完整 State 快照
stream() 在 ReAct Agent 里特别有用：
第八课的循环可能转好几轮，用 stream() 可以实时看到每一轮 agent 在想什么、tools 执行了哪个工具，而不是等最终答案出来才知道发生了什么。
踩坑记录
坑 1：stream() 返回的是迭代器，不是 dict

graph.stream(...) 返回一个生成器，必须用 for chunk in ... 来消费，不能直接取 ["messages"]。
如果只需要最后一个 chunk，可以用 *_, last = graph.stream(...) 或 list(graph.stream(...))。

坑 2：加了 checkpointer 同样需要传 config

第九课加了 MemorySaver 后，stream() 和 invoke() 一样需要带 config。
graph.stream(input, config=config, stream_mode="updates")

11
Human-in-the-loop：interrupt()
在节点内部暂停 Graph，等人类决定后再继续
为什么需要人类介入

前十课的 Graph 一旦启动就自动跑完，没有地方可以让人类审查中间结果。
实际场景中经常需要：

  • LLM 生成了一个操作方案 → 人类确认后再执行
  • Agent 准备调用一个危险工具 → 人类批准才能调用
  • 需要人类补充信息 → 等待用户输入后继续

LangGraph 的解决方案是 interrupt()：在节点内部调用，Graph 立刻暂停，把值传出去，等 Command(resume=...) 进来后从断点继续。

新概念 1：interrupt()

在任意节点函数内部调用 interrupt(value)，Graph 立刻暂停：

from langgraph.types import interrupt

def review_node(state: State) -> dict:
    print(state["plan"])  # 先做完该做的事

    # ★ 在这里暂停，把提示信息传给调用方
    decision = interrupt("请输入 yes 批准，其他取消：")

    # Graph 恢复后，decision 拿到的是 Command(resume=...) 传入的值
    approved = decision.strip().lower() == "yes"
    return {"approved": approved}
interrupt() 传入的参数（这里是字符串 "请输入..."）会出现在 Graph 暂停时的返回值里，用来告诉调用方"我暂停在哪里、需要什么信息"。
新概念 2：Command(resume=...)

人类做出决定后，把 Command(resume=值) 作为输入传给 invoke()，Graph 从断点继续：

from langgraph.types import Command

config = {"configurable": {"thread_id": "review-1"}}

# 第1次：跑到 interrupt() 暂停
result = graph.invoke({"question": "如何读取 CSV？"}, config=config)
# result 包含暂停信息，Graph 没有到达 END

# 人类输入决定后...
human_input = "yes"

# 第2次：传入 Command 恢复，resume 的值会成为 interrupt() 的返回值
result = graph.invoke(Command(resume=human_input), config=config)
注意：第2次 invoke() 的第一个参数不再是 State dict，而是 Command(resume=...)。
必须用相同的 thread_id，这样 checkpointer 才能找到之前暂停的位置。
执行流程图
第1次 invoke({"question": "..."}, config)
  │
  ▼
plan_node
  LLM 生成方案，存入 state["plan"]
  │
  ▼
review_node
  打印方案
  调用 interrupt("请输入 yes 批准...")
  ┌─────────────────────────────┐
  │  Graph 暂停                  │
  │  State 保存到 checkpoint     │
  │  invoke() 返回               │
  └─────────────────────────────┘

       人类查看方案，输入决定

第2次 invoke(Command(resume="yes"), config)
  │
  ▼  从 checkpoint 恢复，回到 review_node 的 interrupt() 处
  interrupt() 返回 "yes"
  approved = True
  review_node 返回 {"approved": True}
  │
  ▼
execute_node
  state["approved"] 为 True → 执行方案
  │
  ▼
END
完整代码（11_interrupt.py）
from langgraph.types import Command, interrupt
from langgraph.checkpoint.memory import MemorySaver

class State(TypedDict):
    question: str
    plan: str
    approved: bool

def plan_node(state: State) -> dict:
    response = llm.invoke([...])
    return {"plan": response.content}

def review_node(state: State) -> dict:
    print(state["plan"])
    decision = interrupt("请输入 yes 批准，其他取消：")
    return {"approved": decision.strip().lower() == "yes"}

def execute_node(state: State) -> dict:
    if state["approved"]:
        print("✓ 方案已批准，开始执行...")
    else:
        print("✗ 方案被拒绝。")
    return {}

# Graph：线性结构，和第三课一样
builder = StateGraph(State)
builder.add_node("plan_node",    plan_node)
builder.add_node("review_node",  review_node)
builder.add_node("execute_node", execute_node)
builder.add_edge(START, "plan_node")
builder.add_edge("plan_node",    "review_node")
builder.add_edge("review_node",  "execute_node")
builder.add_edge("execute_node", END)

graph = builder.compile(checkpointer=MemorySaver())

config = {"configurable": {"thread_id": "review-1"}}

# 第1次：跑到 interrupt() 暂停
graph.invoke({"question": "如何用 Python 读取 CSV？"}, config=config)

# 人类输入
human_input = input("请输入你的决定：")

# 第2次：传入 Command 恢复
graph.invoke(Command(resume=human_input), config=config)
知识点总结
概念 作用 来自
interrupt(value) 在节点内暂停 Graph，把 value 传给调用方 langgraph.types
Command(resume=v) 恢复暂停的 Graph，v 成为 interrupt() 的返回值 langgraph.types
MemorySaver（必须） 保存暂停时的 State，恢复时找到断点位置 第九课
interrupt() vs Python input()：
input() 阻塞终端，只能在命令行用。
interrupt() 暂停 Graph，可以跨进程、跨请求、跨时间恢复——Web 应用里用户点按钮就能恢复，不需要保持连接。
踩坑记录
坑 1：没有 checkpointer 时调用 interrupt() 会报错

interrupt() 依赖 checkpointer 保存暂停点，如果 compile() 没传 checkpointer，运行时会抛异常。
必须：graph = builder.compile(checkpointer=MemorySaver())

坑 2：第2次 invoke 的第一个参数是 Command，不是 State dict

恢复时传的是 Command(resume=...)，不是 {"question": ...}。
如果第2次还传 State dict，Graph 会重新从头开始执行，而不是从断点继续。

坑 3：两次 invoke 必须用相同的 thread_id

checkpointer 靠 thread_id 找到暂停的 State。如果 thread_id 不同，找不到断点，Graph 会报错。

12
外部访问 State：get_state() + update_state()
暂停时查看并修改 Graph 的状态，再决定如何继续
为什么需要外部访问

第十一课的 interrupt() 让 Graph 暂停，但只能做一件事：传入 Command(resume=...) 继续。

现实场景往往更复杂：

  • 暂停后想先看一眼 LLM 生成了什么内容，再决定要不要批准
  • LLM 的方案不对，想直接改掉再继续，而不是重新跑一遍
  • 调试时想手动设置 State 的某个字段，测试后续节点的行为

这两个方法解决这些问题：

方法 作用
graph.get_state(config) 读取当前 State 快照（所有字段值、下一步节点）
graph.update_state(config, values) 从外部直接修改 State 字段，不经过任何节点
get_state()：读取当前 State
snapshot = graph.get_state(config)

# 当前所有字段的值
print(snapshot.values)
# → {"question": "...", "plan": "...", "approved": False}

# 下一步将运行的节点
print(snapshot.next)
# → ("review_node",)   ← 暂停在 review_node 里
# → ()                 ← Graph 已结束

get_state() 返回的是 StateSnapshot 对象，常用的属性：

属性 含义
.values State 的所有字段值（dict）
.next 下一步要运行的节点名 tuple，结束后为空 ()
.tasks 当前挂起任务列表（interrupt 时可看到暂停信息）
update_state()：从外部修改 State
# 只修改 plan 字段，其他字段不变
graph.update_state(
    config,
    {"plan": "【人工修改】1) 加索引 2) 避免 SELECT *"},
)

# 验证修改生效
snapshot = graph.get_state(config)
print(snapshot.values["plan"])
# → "【人工修改】1) 加索引 2) 避免 SELECT *"

之后再 invoke(Command(resume="yes"))，execute_node 看到的就是修改后的 plan。

update_state() 直接写入 checkpointer，不执行任何节点。
只更新你指定的字段，其他字段保持原值不变。
完整使用流程（12_state_ops.py）
config = {"configurable": {"thread_id": "ops-demo"}}

# 第1步：启动 Graph，跑到 interrupt() 暂停
graph.invoke({"question": "如何优化慢查询 SQL？"}, config=config)

# 第2步：查看 LLM 生成的方案
snapshot = graph.get_state(config)
print("当前 plan：", snapshot.values["plan"])
print("下一步节点：", snapshot.next)   # ("review_node",)

# 第3步：发现方案不满意，直接改掉
graph.update_state(
    config,
    {"plan": "【人工修改】使用索引优化..."},
)

# 第4步：批准，继续执行（execute_node 拿到的是修改后的 plan）
graph.invoke(Command(resume="yes"), config=config)
知识点总结
概念 说明
graph.get_state(config) 返回 StateSnapshot，读取当前 State 值和执行位置
snapshot.values State 所有字段的当前值
snapshot.next 下一步节点 tuple；() 表示已结束
graph.update_state(config, values) 直接写入指定字段，不经过节点，不触发执行
第九课到第十二课的联系：
第9课 MemorySaver → State 可以跨调用持久化
第11课 interrupt() → Graph 可以暂停等人类
第12课 get/update_state → 暂停时可以检查和修改 State
三者叠加 = 完整的 Human-in-the-loop 工作流
踩坑记录
坑 1：update_state() 不会触发节点执行

调用 update_state() 只是写入 checkpointer，不会触发任何节点运行。
改完之后还需要 invoke(Command(resume=...)) 才能继续执行 Graph。

坑 2：对 messages 字段用 update_state() 要注意 reducer

如果 State 里有 Annotated[list[BaseMessage], add_messages] 字段，
update_state(config, {"messages": [新消息]}) 会追加而不是覆盖，因为 reducer 是 add_messages。
这和普通字段（直接覆盖）行为不同，要注意。

坑 3：没有 checkpointer 时两个方法都不能用

get_state() 和 update_state() 都依赖 checkpointer 存取数据。
如果 compile() 没传 checkpointer，调用时会报错。

MessagesState：内置消息状态
省掉每课都要重复写的 Annotated[list[BaseMessage], add_messages]
解决什么问题

这一课是对第六课的简化。第六课故意用完整写法，是为了让你看清楚 reducer 的工作原理。理解之后，LangGraph 提供了内置的 MessagesState 把这段模板打包好，不需要每次都重复写。

从第六课起，每次定义 State 都要写同一段 5 行模板。第十三课用 MessagesState 代替它，效果完全相同。

❌ 旧写法（第六课起，每课重复）

from langchain_core.messages import BaseMessage
from langgraph.graph.message import add_messages

class State(TypedDict):
    messages: Annotated[
        list[BaseMessage],
        add_messages
    ]

builder = StateGraph(State)
✅ 新写法（第十三课起推荐）
from langgraph.graph import MessagesState




# 内置好了，直接用
# （内部就是上面那 5 行）

两种写法效果完全一样，节点代码不需要改一个字。

用法 1：直接使用 MessagesState

State 只需要 messages 字段时，直接传给 StateGraph：

builder = StateGraph(MessagesState)

def llm_node(state: MessagesState) -> dict:
    response = llm.invoke(state["messages"])
    return {"messages": [response]}   # 追加语义，和第六课完全一样

用法 2：继承 MessagesState，添加额外字段

需要 messages 之外的字段时，继承后添加：

class MyState(MessagesState):
    topic: str    # 普通字段，覆盖语义
    count: int    # 普通字段，覆盖语义

# messages 自动继承（仍是 add_messages 追加语义）
# topic、count 是普通字段（赋值覆盖）

builder = StateGraph(MyState)
继承后 messages 字段的 reducer 不变，仍然是追加。额外字段是普通覆盖语义，和第一课的 answer、question 字段一样。

14
create_react_agent：一行代码建好 ReAct Agent
第八课的 7 步手动流程，现在用一行函数完成
解决什么问题

这一课是对第八课的简化。第八课故意手动搭了完整的 ReAct Agent，是为了让你理解每一步在做什么：ToolNode 怎么执行工具、tools_condition 怎么路由、循环怎么形成。理解之后，langchain 提供了 create_agent 把这 7 步打包成一行。

版本变化： 旧版 langgraph.prebuilt.create_react_agent 在 LangGraph 1.x 中已废弃。 新版是 from langchain.agents import create_agent（需要安装 langchain 主包）。 本课使用新版。参数变化：旧版用 prompt=，新版用 system_prompt=。
为什么这个快捷方式从 LangGraph 移到了 LangChain？

这背后是两个包的职责分工演变：

时间 发生了什么
LangGraph 早期 create_react_agent 在 langgraph.prebuilt 里，是 LangGraph 原生的快捷方式
LangGraph 1.x LangGraph 专注做"图"的底层基础能力，把"高层封装"移交给 LangChain 维护
现在 langchain.agents.create_agent 是它的新家，底层仍然是 LangGraph，但入口在 LangChain 侧

现在两个包的定位更清晰了：

历史快照：get_state_history()
查看 Graph 每一步的状态变化，随时回到过去重新执行
和 get_state() 的区别

第十二课的 get_state() 只能看当前这一刻的 State。
get_state_history() 返回这个 thread 里每一步执行后保存的所有快照，从最新到最旧。

方法 返回 用途
get_state(config) 当前快照（1个） 查看现在是什么状态
get_state_history(config) 所有历史快照（迭代器） 调试、时光回溯
前置条件同第九、十二课：compile(checkpointer=MemorySaver())，checkpointer 存的就是这些快照。
基本用法
history = list(graph.get_state_history(config))

# 对于一个 3 节点 Graph（llm_node → count_node → print_node），快照从新到旧：
# 快照 0：next=()               ← Graph 结束后
# 快照 1：next=("print_node",)  ← count_node 跑完
# 快照 2：next=("count_node",)  ← llm_node 跑完
# 快照 3：next=("llm_node",)    ← 刚开始
# 快照 4：next=("__start__",)   ← 初始输入

for snapshot in history:
    print(snapshot.next, snapshot.values)

每个 snapshot 是 StateSnapshot，字段和第十二课一样：.values、.next、.config。

时光回溯：从某个历史快照重新执行

找到目标快照后，用它的 .config（内含 checkpoint_id）重新 invoke，Graph 从那个节点接着跑：

# 找到"下一步是 count_node"的快照（llm_node 刚跑完的那一刻）
target = None
for snapshot in graph.get_state_history(config):
    if snapshot.next == ("count_node",):
        target = snapshot
        break

# 用 target.config 重新 invoke：从 count_node 开始，跳过 llm_node
result = graph.invoke(None, config=target.config)
典型场景：
LLM 生成的回答不满意 → 找到 llm_node 之前的快照 → 重新 invoke
→ LLM 重新生成（有随机性，结果可能不同），后续节点重新跑

注意传 None 而不是 State dict，表示"不新增输入，从快照恢复并继续"。

回滚时哪些字段有值、哪些被清掉？

以回滚到"llm_node 刚跑完、count_node 还没开始"的快照为例：

# 回滚后恢复的 State：
{
    "question": "什么是机器学习？",  # ← 有值（llm_node 之前就有）
    "answer":   "机器学习是...",     # ← 有值（llm_node 已写入）
    "word_count": 0,                   # ← 回滚掉了（count_node 还没跑）
}

count_node 重新执行时，return {"word_count": len(state["answer"])} 是这次重新计算出来的新值，不是第一次的结果。
简单说：快照那一刻之后写入的字段全部回滚，之前写入的字段保留。

完整代码（15_state_history.py）
config = {"configurable": {"thread_id": "history-demo"}}

# 执行一次
graph.invoke({"question": "什么是机器学习？"}, config=config)

# 查看所有历史快照

    print(f"快照 {i}：next={snapshot.next}")
    print(f"         values={snapshot.values}")

# 时光回溯：找到 llm_node 刚跑完的快照，从 count_node 重新跑
for snapshot in graph.get_state_history(config):
    if snapshot.next == ("count_node",):
        result = graph.invoke(None, config=snapshot.config)
        print("重新执行完毕，word_count =", result["word_count"])
        break

snapshot.config 含 checkpoint_id，传给 invoke 可从该快照恢复执行

Subgraph：把 Graph 当节点用
编译好的 Graph 可以直接作为另一个 Graph 的节点
什么是子图

前十五课的节点都是 Python 函数。本课引入第二种节点类型：编译好的 Graph。

subgraph = sub_builder.compile()        # 编译成 Graph
outer_builder.add_node("qa", subgraph)  # ★ 直接当节点传入

外层 Graph 调用到 "qa" 这个节点时，实际上是把 subgraph 完整跑一遍，然后把结果写回外层 State。

用途：把复杂逻辑拆成独立的子图，外层 Graph 像调用函数一样使用它，内部细节对外层透明。
State 通信规则：同名字段自动传递

外层 State 和子图 State 通过同名字段交换数据：

方向 规则
外层 → 子图 两边 State 都有的字段，自动传入子图
子图 → 外层 两边 State 都有的字段，子图结果自动写回外层
只有外层有的字段 子图看不到，子图执行后也不会被清掉
只有子图有的字段 不会出现在外层 State 里

本课的例子：

class SubState(TypedDict):
    question: str    # 共有 → 外层传入
    answer: str      # 共有 → 子图写回外层
    word_count: int  # 共有 → 子图写回外层

class OuterState(TypedDict):
    question: str
    answer: str
    word_count: int
    label: str       # 只有外层有 → 子图不知道，不会被清掉

子图作节点 compiled_graph 可以直接传给 add_node()

坑 1：子图字段名和外层对不上，数据传不进去

必须先 subgraph = sub_builder.compile()，再 add_node("name", subgraph)。

Multi-Agent：Supervisor 模式
用 LLM 决定把任务路由给哪个子 Agent
什么是 Supervisor 模式

把前面学过的积木拼在一起：

  • 第4课 条件路由 → Supervisor 根据 LLM 决策路由
  • 第8课 ReAct Agent → 每个子 Agent 内部是完整的工具调用循环
  • 第16课 子图当节点 → 每个子 Agent 是一个编译好的子图

新概念（只有这一个）：Supervisor 模式

  • Supervisor 节点：LLM 读取任务，输出下一步去哪个 Agent
  • 子 Agent 跑完后结果写回共享 State，再交还给 Supervisor
  • Supervisor 再次判断：继续分配任务 or 结束

Supervisor 是唯一的"决策者"，子 Agent 只负责执行。

架构图
START → supervisor → math_agent   → supervisor → ...
                  ↘ writer_agent  → supervisor → ...
                  ↘ END（Supervisor 认为任务完成）
角色 实现方式 职责
Supervisor 普通 LLM 节点 + 条件路由 决策、调度
math_agent 编译子图（ReAct + 工具） 数学计算
writer_agent 编译子图（纯 LLM） 文字写作

Send API：并行执行（Fan-out / Map-Reduce）
把一个任务拆成多份，同时跑多个节点实例
什么是 Send

前面所有课的路由函数都返回一个字符串（下一个节点名），每次只走一条路，串行执行。

Send 让路由函数返回一个列表，同时触发多个节点实例并行跑：

from langgraph.types import Send

def fan_out(state: State) -> list[Send]:
    return [
        Send("worker", {"question": q, "answers": []})
        for q in state["questions"]
    ]
# 3 个问题 → 同时跑 3 个 worker 实例，互不等待

💡 这也是为什么 snapshot.next 是 tuple 而不是字符串
普通路由时：next = ("summarize",) — 只有一个节点
并行路由时：next = ("worker", "worker", "worker") — 同时有多个
设计成 tuple 就是为了统一处理这两种情况。

Reducer 的作用

3 个 worker 同时往 answers 字段写结果，如果没有 Reducer 会互相覆盖，只剩最后一个。

用 Annotated + operator.add 声明 Reducer，告诉 LangGraph "把每次写入的列表拼接进去"：

import operator
from typing import Annotated

class State(TypedDict):
    questions: list[str]
    answers: Annotated[list[str], operator.add]  # ← Reducer
    #         每个 worker 返回 ["答案X"]，自动 add 拼接，不覆盖

class WorkerState(TypedDict):
    question: str
    answers: Annotated[list[str], operator.add]  # worker 也要声明

这和第13课 MessagesState 里的 add_messages 是同一个道理 —— 都是 Reducer。

完整代码（18_send_api.py）
class State(TypedDict):
    questions: list[str]
    answers: Annotated[list[str], operator.add]

class WorkerState(TypedDict):
    question: str
    answers: Annotated[list[str], operator.add]

# Fan-out：拆成多个 Send
def fan_out(state: State) -> list[Send]:
    return [
        Send("worker", {"question": q, "answers": []})
        for q in state["questions"]
    ]

# Worker：处理单个问题
def worker(state: WorkerState) -> dict:
    response = llm.invoke([
        SystemMessage(content="用一句话简洁回答。"),
        HumanMessage(content=state["question"]),
    ])
    return {"answers": [f"Q: {state['question']}\nA: {response.content}"]}

# 汇总节点：所有 worker 完成后执行
def summarize(state: State) -> dict:
    for i, ans in enumerate(state["answers"], 1):
        print(f"{i}. {ans}")
    return {}

builder = StateGraph(State)
builder.add_node("worker",    worker)
builder.add_node("summarize", summarize)

builder.add_conditional_edges(START, fan_out)  # ← 路由函数返回 Send 列表
builder.add_edge("worker",    "summarize")
builder.add_edge("summarize", END)

graph = builder.compile()

graph.invoke({
    "questions": ["什么是机器学习？", "什么是深度学习？", "什么是强化学习？"],
    "answers": [],
})
执行流程
START
  ↓ fan_out() 返回 [Send("worker",q1), Send("worker",q2), Send("worker",q3)]
worker(q1)   worker(q2)   worker(q3)   ← 3 个实例同时并行跑
  ↓               ↓            ↓
每个返回 {"answers": ["Q:...\nA:..."]}
Reducer(operator.add) 把 3 个列表合并成一个 answers
  ↓
summarize()   ← 等所有 worker 完成后才执行
  ↓
END
知识点总结
知识点 说明
Send(node, state) 创建一个并行任务，路由函数返回 Send 列表即可并行
Annotated[list, operator.add] Reducer，并行节点写同一字段时拼接而不是覆盖
snapshot.next 是 tuple 并行时多个节点同时在 next 里，所以设计成 tuple
汇聚（Fan-in） 所有并行 worker 完成后，自动继续执行下一个节点

MemorySaver vs SqliteSaver
MemorySaver SqliteSaver

适合场景 调试、查看 State 变化 前端打字机效果、实时 UI
完整代码（20_astream_events.py）
import asyncio
from langchain_openai import ChatOpenAI

# ★ 开启 LLM 流式输出
llm = ChatOpenAI(..., streaming=True)

async def stream_tokens():
    async for event in graph.astream_events(
        {"messages": [HumanMessage(content="用三句话介绍一下 Python 语言")]},
        version="v2",   # 固定写 v2
    ):
        # 只关心 LLM 输出 token 的事件
        if event["event"] == "on_chat_model_stream":
            token = event["data"]["chunk"].content
            if token:
                print(token, end="", flush=True)  # 逐 token 打印

asyncio.run(stream_tokens())

关键三步：
1. ChatOpenAI(streaming=True) — LLM 开启流式
2. async for event in graph.astream_events(..., version="v2") — 异步迭代事件
3. 过滤 event["event"] == "on_chat_model_stream" — 只取 token 事件

常用事件类型

on_chain_end Graph / 节点执行完毕
on_chat_model_start LLM 开始生成

on_chat_model_end LLM 生成完毕

# 节点通过 *, store: BaseStore 参数注入
def chat_node(state: State, *, store: BaseStore) -> dict:
    user_id  = state["user_id"]
    namespace = ("users", user_id)   # 命名空间，元组表示层级

    # 读取该用户的历史记忆
    memory_item = store.get(namespace, "profile")
    memory_text = f"已知用户信息：{memory_item.value}" if memory_item else ""

    response = llm.invoke([
        SystemMessage(content=f"你是一个助手。{memory_text}"),
        *state["messages"],
    ])

    # 把新信息写入 store（所有 thread 共享）
    store.put(namespace, "profile", {"name": "小明", "hobby": "篮球"})

    return {"messages": state["messages"] + [response]}

graph.invoke(
    {"user_id": "user-001", "messages": [HumanMessage(content="你好，我叫小明，我喜欢打篮球。")]},
    config=config_a,

# ── 第二次对话（thread-B，全新 thread）────────────────
config_b = {"configurable": {"thread_id": "thread-B"}}

)

方法 说明

自定义 Reducer：控制 State 字段如何合并
写任意 (old, new) → merged 函数，挂到 Annotated 上即可
Reducer 的本质

每次节点返回新值时，LangGraph 调用 Reducer：

merged = reducer(state 中的旧值, 节点返回的新值)
# 然后把 merged 写回 State

没有 Reducer → 新值直接覆盖旧值
有 Reducer → 由你决定怎么合并

def my_reducer(old, new):
    ...合并逻辑...
    return merged

class State(TypedDict):
    field: Annotated[类型, my_reducer]   # 挂上去就生效

已学过的内置 Reducer：
  add_messages（第13课）— 追加消息，重复 id 会更新
  operator.add（第18课）— 列表拼接，等价于 old + new
  自定义（本课）— 任意 (old, new) → merged

三种 Reducer 示例
# 1. 滑动窗口：只保留最新 3 条
def keep_last_3(old: list, new: list) -> list:
    return (old + new)[-3:]

# 2. 合并字典：新 key 覆盖旧 key，旧有而新没有的保留
def merge_dict(old: dict, new: dict) -> dict:
    return {**old, **new}

# 3. 去重列表：已存在的不重复添加
def unique_append(old: list, new: list) -> list:
    result = list(old)
    for item in new:
        if item not in result:
            result.append(item)
    return result
Reducer 旧值 新值 结果
keep_last_3 ["a","b"] ["c","d","e"] ["c","d","e"]
merge_dict {"x":1,"y":2} {"y":9,"z":3} {"x":1,"y":9,"z":3}
unique_append ["a","b"] ["b","c"] ["a","b","c"]
完整代码（22_custom_reducer.py）
class State(TypedDict):
    question: str
    logs:     Annotated[list[str], keep_last_3]   # 滑动窗口
    profile:  Annotated[dict,      merge_dict]     # 字典合并
    tags:     Annotated[list[str], unique_append]  # 去重

def node_b(state: State) -> dict:
    return {
        "logs":    ["node_b 执行", "LLM 回答..."],
        "profile": {"step": "b", "answer_len": 42},
        "tags":    ["langgraph", "llm", "node_b"],  # "langgraph" 已存在，不重复
    }

result = graph.invoke({
    "question": "什么是人工智能？",
    "logs": [], "profile": {}, "tags": [],
})
# logs    → 只保留最新3条
# profile → 各节点的 key 合并成一个 dict
# tags    → 去重后的标签列表
知识点总结
知识点 说明
Reducer 签名 def reducer(old, new) -> merged，两个参数固定
挂载方式 Annotated[类型, reducer函数]
无 Reducer 节点返回的新值直接覆盖旧值
有 Reducer LangGraph 自动调用 reducer(旧值, 新值)，结果写回 State

)
